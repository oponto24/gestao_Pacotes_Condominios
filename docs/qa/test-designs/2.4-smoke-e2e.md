## Cenários (Given / When / Then)

### S2 — Criar morador principal (caminho feliz)

**Cobre:** AC5, AC9, AC10, AC11

- **Given** estou em `/admin/moradores` e clico em "Novo morador"
- **When** preencho `nome="Ana Silva"`, `telefone="(11) 99999-1111"`, `email="ana@test.com"`, seleciono uma unidade no dropdown, marco ☑️ "Contato principal" e clico Salvar
- **Then** Sheet fecha, toast "Morador criado com sucesso" aparece, e a linha aparece na lista com badge "Principal" ⭐ e telefone em formato E.164 (`+5511999991111`)

**Critério aprovação:**
- Console: zero erros JS
- Network: `POST /api/admin/moradores` → `201`
- DB: `SELECT nome_normalizado FROM moradores WHERE telefone='+5511999991111'` retorna `"ana silva"` (lowercase + sem acento)

---

### S3 — Invariante 1-principal-por-unidade via UI

**Cobre:** AC5, AC11 — caminho **crítico** identificado pelo gate

- **Given** já existe Ana como principal na Unidade X (S2 ok)
- **When** crio um 2º morador "Bruno" na **mesma** Unidade X também marcado como principal
- **Then** Bruno aparece como principal e Ana **deixa** de ser principal automaticamente (badge "Principal" some da Ana)

**Critério aprovação:**
- DB: `SELECT nome, is_principal FROM moradores WHERE unidade_id='<X>'` retorna exatamente 1 linha com `is_principal=true`
- UI: refresh da lista confirma estado consistente

---

### S4 — Telefone duplicado bloqueia (UNIQUE composite)

**Cobre:** AC2, AC5

- **Given** Ana com telefone `+5511999991111` existe
- **When** tento criar outro morador com mesmo telefone
- **Then** Sheet permanece aberto, toast de erro "Telefone já cadastrado" aparece, formulário NÃO é submetido com sucesso

**Critério aprovação:**
- Network: `POST /api/admin/moradores` → `409 Conflict`
- DB: contagem por telefone permanece 1

---

### S5 — Editar nome recalcula `nome_normalizado`

**Cobre:** AC1, AC7

- **Given** Ana ("Ana Silva", `nome_normalizado='ana silva'`) existe
- **When** abro Sheet de edição, mudo nome para "Ânã  Sîlvâ" (com acentos e espaço duplo) e salvo
- **Then** lista atualiza com novo nome visual, e DB tem `nome_normalizado='ana silva'` (espaço colapsado, acentos removidos)

**Critério aprovação:**
- Network: `PATCH /api/admin/moradores/<id>` → `200`
- DB: `SELECT nome, nome_normalizado FROM moradores WHERE id='<id>'` confirma derivação

---

### S6 — PATCH não aceita `unidade_id` (escape silencioso)

**Cobre:** AC7 — Security: defesa contra escape de tenant via campo de FK

- **Given** Ana está na Unidade X
- **When** abro Sheet de edição (modo edit) e observo que o select de unidade **não aparece** (apenas em create)
- **Then** UX confirma "mover requer recriar"

**Critério aprovação:**
- DOM: select de unidade ausente no modo edit
- Inspect DevTools: confirmar que campo `unidade_id` não existe no formulário de edição

---

### S7 — Soft delete LGPD via UI

**Cobre:** AC8 — caminho crítico

- **Given** Ana existe e está como principal
- **When** clico "Arquivar" no menu de ações da Ana e confirmo
- **Then** Ana some da lista padrão, toast "Morador arquivado" aparece

**Critério aprovação:**
- Network: `DELETE /api/admin/moradores/<id>` → `204`
- DB: `SELECT deleted_at, ativo, is_principal FROM moradores WHERE id='<id>'` retorna `deleted_at IS NOT NULL`, `ativo=false`, `is_principal=false`
- Registro **nunca** é fisicamente deletado (`SELECT COUNT(*)` permanece igual)

---

### S8 — Filtro "Mostrar arquivados" exibe arquivados

**Cobre:** AC4, AC10

- **Given** Ana está arquivada (S7)
- **When** ativo o toggle "Mostrar arquivados" na lista
- **Then** Ana reaparece na lista com badge "Arquivado" e ação "Restaurar" disponível

**Critério aprovação:**
- Network: `GET /api/admin/moradores?include_arquivados=true` → inclui Ana
- UI: badge visual de arquivado distinto do ativo

---

### S9 — Restore zera `deleted_at` mas NÃO restaura `is_principal`

**Cobre:** AC8

- **Given** Ana está arquivada
- **When** clico "Restaurar" na Ana
- **Then** Ana volta para lista padrão como morador ativo, **sem** badge principal

**Critério aprovação:**
- Network: `POST /api/admin/moradores/<id>/restore` → `200`
- DB: `deleted_at IS NULL`, `ativo=true`, `is_principal=false`
- Justificativa: admin precisa re-marcar principal explicitamente (evita conflito silencioso)

---

### S10 — Validação Zod no cliente bloqueia submit

**Cobre:** AC2, AC11

- **Given** Sheet aberto em modo create
- **When** tento submeter com (a) nome curto (1 char), (b) telefone inválido `"abc"`, (c) email malformado `"foo"`
- **Then** mensagens de erro inline aparecem em cada campo, submit não é enviado para API

**Critério aprovação:**
- Network: zero requests `POST /api/admin/moradores` durante tentativas inválidas
- UX: foco volta para o primeiro campo com erro

---

### S11 — Mobile drawer + acessibilidade do Select

**Cobre:** AC9, AC10 — caminho que tests não cobrem

- **Given** viewport mobile (DevTools → 375px)
- **When** abro Sheet, navego até o select de unidade usando **somente teclado** (Tab + setas + Enter), e tento fechar com ESC
- **Then** dropdown abre, navega entre opções com ↑↓, seleção com Enter, ESC fecha tanto o dropdown quanto o Sheet

**Critério aprovação:**
- Acessibilidade: todos os elementos interativos alcançáveis via teclado
- Visual: layout não quebra em 375px (Sheet vira drawer fullscreen)
- Console: zero warnings de acessibilidade Radix
