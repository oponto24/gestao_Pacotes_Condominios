## Notas para o @dev

### Dual `DATABASE_URL` — host vs container

Esta é a parte mais sutil da story. O Prisma CLI roda **no host** (não dentro do container) — é como um `psql`. Mas o app Next.js dentro do container precisa conectar usando o nome do service.

**Estratégia escolhida:**
- `.env.local` (host) → `DATABASE_URL=postgresql://app:dev_secret@localhost:5432/gestao_pacotes`
- Compose service `app`/`worker` → `environment: DATABASE_URL=postgresql://app:dev_secret@postgres:5432/gestao_pacotes` (sobrescreve)

Por que: Prisma CLI (`prisma migrate dev`, `prisma studio`, `prisma db seed`) precisa rodar no host pra gerar arquivos no diretório `prisma/migrations/`. O app dentro do container nunca usa `localhost`.

### Decisões pré-aprovadas (não pedir confirmação)

- Versão Prisma 6.x (mais recente estável)
- Logging de queries em dev, só errors em prod (no `db.ts`)
- Migration nome `initial_schema` (não `init` ou `0001`)
- Não rodar `rls_policies.sql` aqui — é da story 1.4

### Pontos de atenção

1. **Não modificar `prisma/schema.prisma`.** Se Prisma CLI sugerir mudanças (ex: formatação), discutir antes — o schema é canônico e foi assinado pela Dara.

2. **`prisma migrate dev` cria a migration E aplica.** Em produção (story 8.4) usaremos `prisma migrate deploy` (só aplica, não gera).

3. **Se a migration falhar:**
   - Cenário comum: enum já existe no banco (de tentativa anterior). Solução: `npx prisma migrate reset --force` → reinicia banco do zero. Isso é seguro em DEV. Em prod NUNCA fazer isso.
   - Logs do Postgres: `docker compose logs postgres | tail -50`

4. **Seed depende de variáveis Clerk vazias OK.** O seed da Dara usa um `clerk_id` placeholder propositalmente (story 1.5 vai sincronizar com Clerk real).

5. **`tsx` em vez de `ts-node`** porque é muito mais rápido e o ecossistema atual prefere.

6. **Hot reload do Next.js + Prisma:** o singleton em `db.ts` (com `globalThis`) é o padrão exato pra evitar warning "Already 10 instances of Prisma Client" durante desenvolvimento.

7. **Se `npm run build` reclamar de Prisma:** é porque Next.js precisa do client gerado. Solução: rodar `npm run prisma:generate` antes do build. Adicionar `"postinstall": "prisma generate"` no `package.json` é uma boa.

### Não fazer agora (out of scope)

- ❌ Aplicar `rls_policies.sql` (story 1.4)
- ❌ Criar role `webhook_worker` BYPASSRLS (story 1.4)
- ❌ Middleware aplicacional de tenant (story 1.6)

## Dev Notes

### Decisões e desvios

**1. Prisma 6 (não 7) — downgrade obrigatório**
`npm install prisma` pegou 7.8.0 inicialmente. Falhou porque Prisma 7 introduziu breaking change: `datasource.url` no schema não é mais suportado, exige migração para `prisma.config.ts`. O schema da Dara foi escrito para Prisma 6. Downgrade pra `prisma@^6 + @prisma/client@^6` (resolveu pra 6.19.3). Quando upgrade pra Prisma 7 for prioridade, vira story dedicada.

**2. Schema da Dara precisou de mini-fix (relações inversas)**
Prisma 6 exige inversa em toda `@relation`. Adicionei em `model Condominio`:
- `moradores: Morador[]`
- `pacote_fotos: PacoteFoto[]`
- `pacote_eventos: PacoteEvento[]`
Não muda nada estrutural — só explicita relações que Prisma usa pra gerar tipos TS. Documentei em SCHEMA.md §9.5.

**3. Diretório de migration pré-existente conflitava**
Dara criou `prisma/migrations/20260506_initial_schema/` antecipadamente com só `rls_policies.sql` (sem `migration.sql`). Prisma viu a pasta e jogou erro P3015. Solução cirúrgica: movi temporariamente o `rls_policies.sql` pra `/tmp`, deixei Prisma criar a migration limpa em `20260506200212_initial_schema/`, depois movi o RLS pra dentro da pasta nova. Convivem agora — RLS será aplicado em story 1.4.

**4. `postinstall: prisma generate` exige schema na imagem**
Stage `deps` do Dockerfile só copiava `package.json` + `package-lock.json`. `npm ci` rodava `postinstall` → `prisma generate` → falhava por falta de schema. Fix: stage `deps` agora copia também `prisma/` antes do `npm ci`. `.dockerignore` ajustado para excluir só `prisma/migrations` e `prisma/seed.ts`, deixando o `schema.prisma` entrar na imagem.

**5. Volume `node_modules` cacheado da story 1.2 quebrou imports**
A imagem nova tem `@prisma/client` mas o volume nomeado `node_modules` vinha da 1.2 (sem). Solução: `docker volume rm gestao-pacotes-dev_node_modules gestao-pacotes-dev_next_cache` antes de subir. Documentei em troubleshooting do `infra/README.md` (já existia "Volume node_modules está corrompido").

**6. Race condition app+worker criando volume vazio simultaneamente**
Ao subir os dois ao mesmo tempo após `volume rm`, ambos tentaram criar `/.bin` no mesmo volume → erro "file exists". Solução: subir um por vez (`up app` → 3s → `up worker`). Não bloqueia em uso normal — só após reset de volume.

**7. Story dizia "12 enums" mas o real é 8**
Recontagem do schema da Dara: UserRole, TamanhoPacote, PacoteStatus, PacoteEventoTipo, WhatsAppMessageStatus, WhatsAppMessageDirection, CodigoMlStatus, Transportadora = 8. Validei o número correto (8) no banco.

**8. Schema atualizado também é commit desta story**
Mudança da relação inversa em `Condominio` é estrutural (afeta TS types) — vai no mesmo commit pra manter coerência migration↔schema.

**9. `package.json#prisma` deprecated warning**
Prisma 6 emite warning "deprecated, migrate to prisma.config.ts" mas ainda funciona. Migração para `prisma.config.ts` quando subir para Prisma 7 (story futura).
