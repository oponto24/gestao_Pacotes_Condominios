## User Story

**Como** porteiro do condomínio,
**eu quero** abrir um app instalável no meu celular com navegação clara entre
"Nova chegada", "Pendentes" e "Retirada",
**para que** eu trabalhe rápido em pé na portaria, com uma mão, sem precisar
abrir o navegador toda vez.

---

## Contexto

**FR-010 (PRD):** "Funcionário deve poder iniciar o registro de pacote pelo
celular (PWA mobile-first)."
**NFR-052 (PRD):** "PWA instalável (Add to Home Screen) com ícone próprio."
**NFR-050:** "Interface da portaria: zero treinamento — operável por qualquer
pessoa em <2 minutos de exposição."

**Esta é a story de fundação da portaria.** Define:
1. **`PortariaLayout`** — template usado por `/chegada`, `/portaria/pendentes`, `/retirada`
2. **`BottomNavBar`** — barra fixa inferior com 3 abas + ícones (mobile-first)
3. **`manifest.json`** — habilita "Add to Home Screen" no celular

Não tem feature de negócio (captura, IA etc.) — isso vem em 3.2+. Aqui só
existe o esqueleto navegável + permissão de role.

**Decisão importante:** **rota `/portaria/*` é restrita a role `porteiro` (e `admin` que tenha permissão).** Super-admin é redirecionado para `/super-admin`. UI da portaria NÃO tem sidebar — apenas bottom nav (mobile-first).

## Notas para o @dev

### Decisões pré-aprovadas

1. **Bottom nav fixo, não tab bar superior:** mobile-first puro. UX consistente com apps mobile que porteiros já usam (WhatsApp, Instagram).
2. **Labels SEMPRE visíveis** (não icon-only): NFR-050 zero treinamento. Ícones isolados confundem usuários menos técnicos.
3. **`requirePorteiro()` aceita `admin` também:** admin pode operar a portaria se necessário (substituir porteiro afastado). Super-admin NÃO opera portaria — supervisiona.
4. **Stubs simples nas 3 rotas:** apenas heading + texto "Em breve". Nada de captura/scan/etc — isso é 3.2+.
5. **Ícones PWA placeholder:** geração simples (PNG vermelho com "P" branco ou similar) é OK pra dev. Documentar em runbook que produção precisa Figma export real.
6. **`src/app/manifest.ts` (Next 14 convention):** preferido sobre `public/manifest.json` estático porque permite TS-typed.
7. **Pattern espelha AdminLayout:** server component carrega ctx + condominio nome → passa pro client component que tem state interativo (bottom nav).
8. **Sem service worker no MVP:** offline NÃO é suportado (NFR-053). Worker offline = scope creep, fica pós-piloto.

### Pontos de atenção

1. **`requirePorteiro` redirect chain:** caminho de auth/role tem que ser determinístico. Sem auth → `/sign-in`. Auth + super_admin → `/super-admin`. Auth + admin/porteiro → permite acesso. Outros roles → 403.
2. **`manifest.ts` requer rebuild Next pra refletir:** depois de criar, `npm run build` precisa rodar pra gerar `manifest.webmanifest`. Em dev, Next regenera automaticamente.
3. **Apple Touch Icon precisa estar em `public/`:** iOS não lê `manifest.json` — usa o `<link rel="apple-touch-icon">`.
4. **`(portaria)` route group:** parênteses fazem Next NÃO incluir no path. URL final é `/chegada`, não `/portaria/chegada`. EXCETO `/portaria/pendentes` que é explícito.
5. **iOS Safari: `apple-mobile-web-app-capable` meta tag:** adicionar no root layout para PWA standalone funcionar bem no iOS.

### Não fazer agora

- PhotoCapture component → 3.2
- BarcodeScannerInput → 3.3
- API POST /api/pacotes → 3.4
- Worker IA → 3.5
- Tela /chegada com formulário real → 3.6+
- Service worker offline → pós-MVP
- Ícone branded por condomínio (white-label) → premium futuro
