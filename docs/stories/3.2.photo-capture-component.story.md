## User Story

**Como** porteiro,
**eu quero** apontar a câmera do celular pra etiqueta do pacote, tirar uma foto
nítida, ver o preview e refazer se ficou ruim,
**para que** a IA receba uma imagem boa pra extrair os dados (sem precisar
digitar manualmente).

## Contexto

**FR-012 (PRD):** "Sistema deve **obrigatoriamente capturar uma foto da
etiqueta** do pacote."
**NFR-051 (PRD):** "Suporte a câmera traseira do celular para bipe e foto."
**UX_SPEC §4.1:** wireframe da tela `/chegada` com 5 estados (câmera off,
ativa, capturada, erro, bipe).

**Esta story entrega APENAS o componente reusável `<PhotoCapture>`.** Não
implementa a tela `/chegada` completa nem chama API — isso é story 3.6.
Foco é state machine + getUserMedia + UI/UX bem cuidada para reuso.

**API esperada:**
```tsx
<PhotoCapture
  onCapture={(blob, dataUrl) => /* handler */}
  onError={(message) => /* handler opcional */}
  preferredFacingMode="environment"  // câmera traseira por padrão
/>
```

Componente devolve `Blob` (pra upload via FormData) + `dataUrl` (pra preview rápido).

## Notas para o @dev

### Decisões pré-aprovadas

1. **`getUserMedia` direto** (sem libs como react-webcam): controle total sobre cleanup + state machine. react-webcam é um wrapper fino que adiciona ~5KB e não traz nada que justifique.
2. **Capture via canvas:** `<video>` + `canvas.drawImage(video, ...)` + `canvas.toBlob('image/jpeg', 0.85)`. Pattern padrão, suporta todos os browsers modernos.
3. **JPEG quality 0.85:** equilíbrio entre qualidade legível pela IA (Claude vision) e tamanho do upload (~150-300KB típico). PNG seria 5x maior, sem ganho visível pra IA.
4. **Bound 1280px no maior lado:** Claude Haiku vision aceita até 8000x8000 mas processa em downsize, e 1280px é resolução suficiente pra texto de etiqueta. Reduz upload + custo de IA.
5. **`facingMode: 'environment'` com fallback:** câmera traseira em mobile (foco automático melhor) + qualquer câmera em desktop.
6. **`useReducer` para state machine:** transições explícitas, fácil de testar. `useState` com 5 estados vira spaghetti.
7. **Vibration + flash visual:** microinteraction polishing, não bloqueante. Se browser não suporta `navigator.vibrate`, simplesmente ignora.
8. **Playground em `/chegada` temporário:** substitui o stub da 3.1 com a integração real. Story 3.6 vai substituir de novo com formulário completo.

### Pontos de atenção

1. **HTTPS obrigatório em produção:** `getUserMedia` só funciona em `https://` (ou localhost). Documentar no runbook que dev usa localhost (OK) e produção precisa SSL configurado (já planejado para story 8.4).
2. **Tracks NÃO param automaticamente** quando componente desmonta — precisa cleanup explícito via `useEffect` return + cleanup quando muda de estado streaming → captured.
3. **iOS Safari quirks:** `playsinline` attribute obrigatório no `<video>` ou ele tenta abrir fullscreen. Adicionar `muted` também (autoplay sem som permitido).
4. **`canvas.toBlob` é assíncrono:** retorna via callback, NÃO via return. Wrappar em Promise se precisar await.
5. **`videoTrack.getSettings()` pode retornar undefined width/height** no Safari antigo. Fallback para `videoElement.videoWidth/videoHeight` (lido após `loadedmetadata` event).
6. **Vibration API quirks:** alguns browsers exigem que seja chamado dentro de gesture user (click handler) — está OK porque captura é após click do botão.
7. **iOS bloqueia `navigator.vibrate`** completamente — não é erro, só não vibra. OK.

### Não fazer agora

- API POST /api/pacotes (upload da foto) → 3.4
- Pipeline IA Claude vision → 3.5
- Tela de chegada completa com formulário → 3.6
- BarcodeScannerInput → 3.3 (story paralela, mesmo nível)
- Editor de imagem (crop, brightness) → não é MVP

## Dev Notes

### Pattern reusado de stories anteriores
- Pattern de hooks de cleanup (useEffect return) — padrão React
- `cn()` helper para classes condicionais
- Tokens Tailwind já estabelecidos (primary, ghost, h-56px botões)

### Nova superfície
- **`getUserMedia` API** — primeiro uso. Documentar permissão em runbook. Será reutilizado pela 3.3 (BarcodeScanner usa o mesmo stream com html5-qrcode).
- **Canvas capture** — primeiro uso. Pode virar util `src/lib/media/capture-frame.ts` se 3.3 também precisar.
- **Vibration + flash visual** — primeiros microinteractions. Pode evoluir para util `src/lib/ux/feedback.ts` em stories futuras.

### Riscos

1. **Permissão negada uma vez = bloqueada:** depois de "Block" no Chrome, user precisa ir em Configurações pra resetar. Documentar no runbook.
2. **iOS Safari standalone (PWA instalado):** `getUserMedia` funciona em PWA standalone, mas nem sempre lembra a permissão entre sessões. Pode pedir várias vezes.
3. **Resolução variável entre dispositivos:** iPhone tira em 4032x3024 nativo, Android pode variar muito. Bound de 1280px normaliza.
4. **MediaStream vazada (memory leak):** se cleanup falhar, câmera fica "ligada" indefinidamente (luz LED acende). Cleanup explícito é crítico.

## Dev Agent Record (2026-05-07)

**Implementação:** @dev (Dex) — componente reusável de captura de foto.

### Decisões e desvios

1. **State machine via `useReducer`** com 5 kinds (`idle`/`requesting`/`streaming`/`captured`/`error`) e 6 actions. Discriminated union TypeScript-typed.
2. **Cleanup centralizado em 3 effects:** (a) effect de request com flag `cancelled`, (b) effect que para stream em qualquer state ≠ streaming/requesting, (c) cleanup global no unmount. Refatorado após teste falhar identificando race condition (cleanup do effect de request rodava mesmo após sucesso → stop prematuro).
3. **`useMemo` para `videoConstraints`** (sugestão @po): objeto literal não recria a cada render, evitando re-execução desnecessária do useEffect.
4. **Fallback de câmera traseira:** se `OverconstrainedError` (desktop sem traseira), tenta `{ video: true }` plain. Outros erros propagam normalmente.
5. **Bound 1280px no maior lado** via `fitInBox` + `canvas.toBlob('image/jpeg', 0.85)`. Tamanho típico ~150-300KB.
6. **`videoTrack.getSettings()` com fallback:** se browser antigo (Safari) não retornar width/height, usa `video.videoWidth/videoHeight`.
7. **Vibration + flash visual** no capture: ambos no `try/catch` ou opcional encadeamento — graceful degradation se browser não suporta.
8. **Privacy disclosure** (sugestão @po): dot vermelho pulsante + "Câmera ligada" no streaming, com `aria-live="polite"`.
9. **`role="alert"` no estado error**: leitor de tela anuncia erro automaticamente.
10. **`debugDownload` prop** (sugestão @po): em dev, renderiza link `<a download>` no captured state. Acelera troubleshooting visual.
11. **Playground em `/chegada`** (story 3.2 temporário): `<ChegadaPlaygroundClient>` mostra Blob size + tipo + dataUrl preview. Será substituído pela tela completa na 3.6.
12. **Test de leak detection completo** (sugestão @po): valida que `track.stop()` é chamado em unmount, transição para captured, e quando entra em error após retake. 3 paths cobertos.
