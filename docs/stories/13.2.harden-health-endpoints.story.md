# Story 13.2 — Proteger health endpoints contra information disclosure

**Epic:** E13 — Hardening UI/UX + Segurança Frontend (P0)
**Status:** Done
**Owner:** @dev (Dex)
**Tipo:** Security fix
**Severidade:** MÉDIA
**Estimativa:** 0.5 dia

---

## User Story

**Como** operador de infraestrutura,
**eu quero** que os health endpoints não exponham versão, uptime ou detalhes de stack,
**para que** atacantes não consigam mapear vulnerabilidades conhecidas.

---

## Contexto

`/api/health` retorna `APP_VERSION`, `uptime_s`, latência de DB/Redis e hostname. `/api/health/db` e `/api/health/redis` retornam detalhes de erro. Essas informações facilitam reconnaissance.

---

## Acceptance Criteria

### AC1 — Health público minimalista
- [x] `/api/health` retorna apenas `{ status: "ok" | "degraded" | "down" }` — sem versão, uptime, ou checks detalhados
- [x] Detalhes completos disponíveis apenas com header `Authorization: Bearer {HEALTH_SECRET}` (novo env var)

### AC2 — Health/db e health/redis protegidos
- [x] `/api/health/db` e `/api/health/redis` retornam 401 sem auth header
- [x] Com auth: retornam dados completos (latência, status, erro)

### AC3 — Erro genérico
- [x] Erros de DB/Redis retornam `{ status: "down" }` sem mensagem de erro detalhada no modo público

### AC4 — Testes
- [x] Test: GET /api/health sem auth retorna apenas status
- [x] Test: GET /api/health com auth retorna detalhes completos
