## User Story

**Como** operador SaaS (Gustavo),
**eu quero** um endpoint `/api/health` que retorne status agregado de db + redis + outros componentes críticos, monitorado por **UptimeRobot** (gratuito, 50 monitors),
**para que** eu seja alertado por e-mail/Telegram em <5 minutos se qualquer componente do sistema cair, sem precisar abrir o app pra verificar.

---

## Contexto

A story 1.3 entregou `/api/health/db` (só checa Postgres). Esta expande pra:
- Endpoint **agregado** `/api/health` (db + redis + uptime + version)
- Endpoint **detalhado** `/api/health/redis` (smoke test do Redis isolado)
- Configuração de **healthcheck Docker** no compose pra app/worker (não só postgres/redis)
- Documentação de **setup UptimeRobot** pro fundador (apontando pra URL pública via cloudflared em dev, ou domínio real em prod)

**Por que monitoramento externo importa:** uptime cron interno detecta crashes, mas só monitor externo detecta network issues, DNS quebrado, certificate expirado, etc. UptimeRobot é gratuito, manda alerta em 5 min, suporta SMS/Telegram/e-mail/Slack.

**Out of scope (vem em stories futuras):**
- Logger estruturado (Pino) → 1.9
- Métricas Prometheus + Grafana → pós-MVP
- APM (Sentry, etc.) → pós-MVP

## Notas para o @dev

### Decisões pré-aprovadas

- **`ioredis` em vez de `redis` (node-redis):** ioredis é o cliente padrão do BullMQ (que vem na 1.8). Evita ter dois clientes diferentes.
- **`/api/health` SEM auth:** monitor externo não tem cookie/JWT. Endpoint público é o padrão da indústria. Não vaza dados sensíveis (só status).
- **Healthcheck `wget`:** alpine tem BusyBox wget por default. Não usar `curl` (não vem por default em alpine, exigiria install extra).
- **Worker healthcheck simples:** worker não tem porta HTTP. Solução: `test: ["CMD", "pgrep", "-f", "node workers/index"]`. Se processo node tá vivo, healthy.

### Pontos de atenção

1. **Timeout é crítico.** Sem timeout, se Postgres travar, o endpoint trava — UptimeRobot vai timeout em 30s e reportar DOWN sem distinguir DB lento de app caído. Implementar com `Promise.race`.

2. **Não logar dados sensíveis no /api/health.** Se DB falhar, não retornar string de erro completa (pode ter password, query, etc.). Só `status: 'down'` + categoria do erro (ex: `connection_refused`).

3. **`uptime_s` via `process.uptime()`** — built-in Node, retorna segundos desde inicio do processo.

4. **`version` via `package.json`** — importar como JSON ou hardcoded por simplicidade. Decidir e documentar.

5. **Redis lazy connect:** `new Redis(url, { lazyConnect: true })`. Não tenta conectar no import — só na primeira query. Evita crash do app se Redis tiver lento pra subir.

6. **Healthcheck app pode falhar nos primeiros segundos** porque Next.js dev compila JIT na primeira requisição. Solução: `start_period: 30s` no compose dá tempo pro Next compilar antes de marcar como unhealthy.

7. **UptimeRobot precisa URL pública.** Em DEV usa cloudflared (mesmo tunnel da Clerk webhook 1.5). Em prod usa o domínio da VPS. Documentar ambos no runbook.

8. **Free tier UptimeRobot:** 50 monitors, intervalo mínimo 5 min. Suficiente pra MVP. Quando passar 50 condomínios e quiser monitor por condomínio, considerar upgrade.

### Não fazer agora

- ❌ Métricas Prometheus / Grafana — pós-MVP
- ❌ APM (Sentry, OpenTelemetry) — pós-MVP
- ❌ Alertas customizados (Telegram bot, Slack webhook) — fundador faz manualmente no UptimeRobot
- ❌ Logger estruturado — vem na 1.9

## Dev Notes

### Decisões e descobertas

**1. `ioredis` instalado com `maxRetriesPerRequest: null`**
Esse é requirement do BullMQ (story 1.8). Documentei já. Não impacta uso comum (apenas afeta retry de comandos individuais Redis).

**2. `lazyConnect: true` no ioredis**
Não conecta no `import` — só na primeira query. Evita crash do app se Redis demorar pra subir. Importante porque `redis.ts` é importado em `health.ts` que é importado em `app/api/health/route.ts` — sem lazy, qualquer carregamento da rota tentaria conectar.

**3. Helper `withTimeout` — Promise.race**
Implementação correta: `clearTimeout` em ambos resolve/reject pra evitar leak. Sem isso, mesmo se Promise resolver antes do timeout, o `setTimeout` continuava no event loop (memory leak em produção sob carga).

**4. `categorizeError` — não vaza connection string**
Mapeia erro pra categoria pública (`timeout`, `connection_refused`, `dns_error`, `auth_failed`, `error`). NÃO retorna `err.message` cru (poderia conter user, password, query). Defesa em endpoint público.

**5. `aggregateStatus` — 3 estados (ok/degraded/down)**
- todos checks ok → `ok`
- todos checks down → `down`
- alguns down → `degraded`

UptimeRobot interpreta 5xx como DOWN — uso 503 pra ambos `degraded` e `down` porque do ponto de vista externo, o app não está plenamente funcional.

**6. `import { version } from '../../../../package.json'`**
Funciona graças a `resolveJsonModule: true` no tsconfig (configurado na story 1.1). Versão fica visível no health pra debug ("qual deploy é esse?").

**7. Worker healthcheck via `pgrep -f "node workers/index"`**
Worker não tem porta HTTP — não dá pra usar wget como o app. `pgrep -f` busca pelo padrão na linha de comando completa. Funciona em alpine (BusyBox) e Linux padrão.

**8. App healthcheck `start_period: 60s` (não 30s da story original)**
Aumentei pra 60s porque Next.js dev compila JIT na primeira requisição — pode demorar mais que 30s no primeiro build do Docker. Em prod (story 8.4) será menor (build estático).

**9. Refatoração do `/api/health/db` (1.3) — DRY**
Endpoint legado agora delega pra `checkDb()` helper. Continua retornando JSON compatível pra retrocompatibilidade. Sem código duplicado.

**10. Test skip pattern revisitado**
Mesma issue da story 1.4 e 1.6: `it.skipIf` é avaliado na coleta. Solução: heurística estática `APP_REACHABLE_STATIC` baseada em env var na coleta + `if (!RUNTIME) return` dentro do teste. Padrão a documentar em ADR pra futuras stories integration.
