## User Story

**Como** admin do condomínio,
**eu quero** cadastrar/editar/desativar as unidades (apartamentos/casas) do meu
condomínio (ex: "Apto 101 Bloco A", "Casa 12"),
**para que** os porteiros possam vincular pacotes a unidades específicas e os
moradores recebam notificações WhatsApp corretamente.

---

## Contexto

**Espelha 1:1 o pattern da 2.2 (Setor)** — mesmo `requireAdmin` + `withTenant` +
Zod + Sheet form + List client. Substitui o stub `/admin/unidades` da story 2.7.

**Modelo `Unidade` (story 1.3):**
- `condominio_id` (FK, RLS scope)
- `identificador` (VARCHAR 50) — "101", "Apto 301-B", "Casa 12"
- `bloco` (VARCHAR 50, opcional) — "A", "Torre 1"
- `observacoes` (VARCHAR 500, opcional)
- `ativo` (boolean, default true)
- **Unique composite:** `(condominio_id, identificador, bloco)`

**Diferenças vs Setor (2.2):**
- 4 campos (vs 3 do Setor)
- Unique tem 3 colunas (inclui `bloco` que pode ser NULL — tratamento especial)
- DELETE precisa validar **2 counts**: `_count.moradores` + `_count.pacotes`
  (Setor só validava pacotes)

**Esta story:**
1. Validators Zod `unidadeCreateSchema` + `unidadeUpdateSchema` + `unidadeListQuerySchema`
2. DB helpers em `src/lib/db/unidade.ts` usando `withTenant()`
3. 5 endpoints REST (mesmo padrão da 2.2)
4. UI substitui stub `/admin/unidades`
5. **Reusa:** `requireAdmin`, `EmptyState`, `Sheet`, `handleApiError`, `errors.ts`, todo o Atlas de UI da 2.1+2.7

**Out of scope:**
- Vínculo com Setor (Unidade NÃO referencia setor — modelo confirmou; setor é onde PACOTE fica, não onde MORADOR mora)
- Importação CSV — story 2.5/2.6 trata
- Bulk operations — futuro
- Audit log — story 8.3
- Filtro por bloco (UI pode adicionar query `?bloco=A` no futuro se virar dor)
- Reordenação manual

## Notas para o @dev

### Decisões pré-aprovadas

- **Espelhar 1:1 o pattern da 2.2.** Toda decisão arquitetural já validada lá.
- **`bloco` opcional + unique composite:** Postgres trata `NULL` como distinto em UNIQUE — duas unidades com `(101, NULL)` colidem; mas `(101, NULL)` e `(101, 'A')` coexistem. Validação Zod aceita string vazia → undefined → coluna NULL.
- **DELETE valida 2 counts** (moradores + pacotes) — UI esconde botão "Excluir" se algum > 0.
- **Lista ordenada por `bloco ASC, identificador ASC`** — agrupa visualmente por bloco.
- **`pageSize` max 200** (vs 100 do Setor) — condomínios grandes podem ter 100+ unidades, melhor permitir lista maior.
- **`<UnidadeForm>` 3 campos** (Identificador, Bloco, Observações) — usar `<Input>` para os 3 (textarea seria over-engineering para observações até 500 chars).

### Pontos de atenção

1. **Unique composite com NULL:** Postgres NÃO considera `(condominio_id, '101', NULL)` igual a outro `(condominio_id, '101', NULL)` — então DUPLICATAS COM BLOCO NULL passariam pela DB. Solução: `findUnidadeByIdentificador` deve fazer query manual considerando NULL com `= IS DISTINCT FROM` ou checar antes via `findFirst({ where: { identificador, bloco }})`. **Atenção especial.**

2. **Test integration de unique composite:** validar tanto `(101, 'A') vs (101, 'B')` (deve coexistir) quanto `(101, NULL) vs (101, NULL)` (deve falhar — depende de como `findUnidadeByIdentificador` trata).

3. **DELETE com fixture de morador:** test integration precisa CRIAR morador antes pra validar 409. Schema já tem o model — apenas criar fixture mínima.

4. **Página `/admin/unidades`** com `pageSize: 100` (vs 200 do schema máximo) — mesmo trade-off da 2.2.

5. **Logger Pino** com `unidade_id` + `condominio_id` (campos consistentes com 2.2).

### Decisões e desvios

1. **`findUnidadeByIdentificador(id, bloco, excludeId)`** trata `bloco: null` explicitamente — Prisma traduz para `IS NULL` no SQL. Sem isso, Postgres deixaria duplicatas-NULL passarem (UNIQUE não trata NULL = NULL).
