## User Story

**Como** admin do condomínio,
**eu quero** subir um CSV com unidades + morador principal de cada uma,
**para que** eu ative um condomínio novo em segundos sem cadastrar uma a uma.

---

## Contexto

**FR-006 (PRD):** importação em massa via CSV de unidades + condômino principal.
**FR-008 (PRD):** validação de telefones BR + sem duplicatas dentro do mesmo condomínio.
**NFR-030 (PRD):** CSV import = condomínio declara ter consentimento LGPD.

Esta story é a **primeira metade** do fluxo: recebe arquivo, faz parse, valida cada linha + detecta duplicatas intra-arquivo, e devolve **um relatório estruturado** (linhas válidas + linhas com erro). **Não persiste nada no banco.**

A story 2.6 vai pegar esse relatório, mostrar preview pra admin (com badges de erro), permitir correção inline OU re-upload, e só então faz o commit transacional.

**Decisão importante:** `name_normalizado` é calculado server-side (mesma estratégia da 2.4). CSV não precisa trazer essa coluna.

## Acceptance Criteria

### AC1 — Schema CSV documentado
- [ ] `docs/runbooks/admin-cadastros.md` ganha seção **"Importação CSV"** com:
  - Cabeçalho obrigatório: `bloco,identificador,morador_nome,morador_telefone,morador_email`
  - Encoding: UTF-8 (com ou sem BOM — parser strip BOM)
  - Separador: `,` (vírgula). Aspas duplas escapam vírgulas internas (RFC 4180).
  - Exemplo de 3 linhas válidas no runbook.
  - **`bloco`** opcional (string vazia = sem bloco)
  - **`morador_email`** opcional (string vazia = sem email)
  - Demais colunas obrigatórias.

### AC2 — Validator Zod (`src/lib/validators/csv-import.ts`)
- [ ] `csvRowSchema` com 5 campos: `bloco?`, `identificador`, `morador_nome`, `morador_telefone`, `morador_email?`
- [ ] Reusa **PHONE_REGEX + normalizePhone** de `_shared.ts` (story 2.4)
- [ ] `morador_telefone` → preprocess para E.164 (mesma lógica da 2.4)
- [ ] `morador_email` aceita string vazia → undefined
- [ ] `morador_nome` min 3, max 200 (mesma da 2.4)
- [ ] `identificador` min 1, max 50 (mesma do validator de Unidade)
- [ ] Tests unit cobrindo: payload mínimo, opcional vazio, telefone inválido rejeitado, nome curto rejeitado.

### AC3 — Parser CSV (`src/lib/csv/parse-import.ts`)
- [ ] Função `parseImportCsv(input: string): ParseResult` síncrona (sem I/O)
- [ ] Strip de BOM no início
- [ ] Detecta cabeçalho ausente → retorna erro `MISSING_HEADER` antes de processar linhas
- [ ] Detecta cabeçalho com colunas faltando → erro `MISSING_COLUMN` listando colunas ausentes
- [ ] Tolera ordem das colunas (mapeia por nome, não por posição)
- [ ] Trim em todos os campos
- [ ] Linhas em branco ignoradas (não contam como erro nem como válida)
- [ ] **Limite duro de 1000 linhas** (proteção DoS) — exceder → erro `TOO_MANY_ROWS`
- [ ] Usa `papaparse` (já é a referência da indústria; ~7KB gz)

### AC4 — Validação por linha + cross-row
- [ ] Cada linha passa pelo `csvRowSchema` → `valid` ou erro estruturado com `linha N: campo: mensagem`
- [ ] **Duplicata intra-arquivo de `(bloco, identificador)`** → ambas as linhas marcadas como erro `DUPLICATE_UNIDADE` (linha X conflita com linha Y)
- [ ] **Duplicata intra-arquivo de `morador_telefone`** → ambas as linhas marcadas como erro `DUPLICATE_TELEFONE`
- [ ] Validação cross-row é após a validação por linha (não rejeita arquivo inteiro só pq tem 1 linha inválida)

### AC5 — Resultado estruturado
- [ ] Retorno do parser:
  ```ts
  type ParseResult =
    | { kind: 'fatal'; code: 'MISSING_HEADER' | 'MISSING_COLUMN' | 'TOO_MANY_ROWS'; details: string }
    | { kind: 'parsed'; valid: ValidRow[]; invalid: InvalidRow[]; totalRows: number }
  ```
- [ ] `ValidRow` carrega os 5 campos já normalizados (telefone E.164, trim aplicado)
- [ ] `InvalidRow` carrega `linha`, `raw` (linha original), `errors: string[]`
- [ ] Ordem preservada (tanto valid quanto invalid em ordem do arquivo)

### AC6 — Endpoint API `POST /api/admin/csv-import/parse`
- [ ] Recebe `multipart/form-data` com campo `file` (apenas .csv ou .txt)
- [ ] `requireAdmin()` (igual aos outros endpoints admin)
- [ ] Tamanho máximo do arquivo: **256KB** (1000 linhas × ~256 bytes/linha)
- [ ] Content-Type aceito: `text/csv`, `text/plain`, `application/vnd.ms-excel` (Excel exporta com esse último)
- [ ] Retorna o `ParseResult` direto como JSON
- [ ] **Não persiste nada** — só responde com o relatório
- [ ] HTTP codes:
  - 200 → parse OK (mesmo que 100% das linhas sejam inválidas — `parsed` com `valid: []`)
  - 400 → arquivo ausente, content-type errado, tamanho excedido, ou `kind: 'fatal'`
  - 401 → sem auth
  - 403 → não-admin

## Notas para o @dev

### Decisões pré-aprovadas
1. **`papaparse` ao invés de parser custom:** lib madura, bem testada contra RFC 4180, ~7KB gz. Evita reinventar parser frágil.
2. **256KB / 1000 linhas como limite:** estimativa conservadora (256 bytes/linha). Ajustável depois se condomínios reais excederem.
3. **Parser síncrono:** parse de 1000 linhas é <50ms. Não precisa stream/worker no MVP.
4. **`sessionStorage` ao invés de URL state:** preview da 2.6 vai ter ~50KB de payload, não cabe na URL.
5. **Mantém `_shared.ts` da 2.4 como fonte da verdade do telefone E.164** — não duplicar regex.
6. **Endpoint só lê e responde** — proibido qualquer `tx.morador.create` ou `tx.unidade.create` aqui.

### Pontos de atenção
1. **BOM (U+FEFF) no início:** Excel salva CSV com BOM por padrão. Strip antes de qualquer parse.
2. **Excel salva como `application/vnd.ms-excel`** mesmo sendo CSV — aceitar esse content-type.
3. **Telefone duplicado intra-arquivo:** marcar **ambas** as linhas como erro (não só a segunda — admin não sabe qual é "a primeira").
4. **`bloco` opcional:** `(bloco=null, identificador="101")` e `(bloco="A", identificador="101")` NÃO são duplicata.
5. **Não normalizar `nome_normalizado` no parser** — a 2.6 vai chamar o helper compartilhado durante o commit.
6. **Erros multi-causais por linha:** uma linha pode ter 3 erros (telefone inválido + nome curto + email malformado). Reportar todos no `errors: string[]`.

### Não fazer agora
- Preview rico com correção inline → 2.6
- Commit transacional no banco → 2.6
- Validação contra unidades JÁ existentes no condomínio → 2.6 (essa é cross-tenant DB, não intra-arquivo)
- Logs de auditoria de import → 8.x (observabilidade)
- CSV de moradores adicionais (não-principais) → fora do escopo MVP

## Dev Notes

### Pattern reusado da 2.4
- `PHONE_REGEX`, `normalizePhone`, `digitsOnly` de `src/lib/validators/_shared.ts`
- `requireAdmin()` em todo endpoint `/api/admin/*`
- `handleApiError` para respostas consistentes
- Validators Zod com `.preprocess` para sanitização

### Nova superfície
- **`src/lib/csv/`** — primeiro diretório de CSV. Provavelmente vai ganhar `format-export.ts` em stories futuras (export CSV de relatórios).
- **`papaparse`** — primeira dependência de parsing. Vale documentar no architecture/ARCHITECTURE.md depois.

### Riscos
1. **Encoding diferente de UTF-8:** se condomínio salvar Excel com Windows-1252, acentos virarão `?`. Mitigação: documentar UTF-8 explícito no runbook + exemplo de "Salvar como UTF-8" no Excel.
2. **Linhas com vírgula no nome ("Maria, Silva"):** RFC 4180 exige aspas. `papaparse` trata. Test cobre.
3. **CSV com "morador_telefone" mas vazio:** validator rejeita por causa do `min(1)`. Decisão: telefone é obrigatório (FR-008). Documentar.
