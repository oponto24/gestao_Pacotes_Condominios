## User Story

**Como** admin do condomínio,
**eu quero** criar/editar/desativar setores (ex: Bloco A, Salão de Festas, Sala de Triagem),
**para que** o porteiro possa classificar pacotes por setor de armazenamento e o
sistema mostre a localização correta na tela de retirada.

---

## Contexto

Story inaugural dos CRUDs **tenant-scoped** do Epic 2. Substitui o stub
`/admin/setores` da story 2.7 pela tela real. Fundação para 2.3 (Unidade) e
2.4 (Morador), que seguirão o mesmo pattern.

**Modelo `Setor` (story 1.3):**
- `condominio_id` (FK, RLS scope)
- `nome` (VARCHAR 100, unique por condomínio)
- `descricao` (VARCHAR 300, opcional)
- `capacidade` (Int, opcional — nº máximo de pacotes simultâneos)
- `ativo` (boolean, default true)

**Diferença fundamental vs. 2.1 (Condominio):**
- Condominio é **cross-tenant** (super_admin) — usa `db.condominio` direto
- Setor é **tenant-scoped** (admin do próprio condomínio) — DEVE usar `withTenant()` da story 1.6 para aplicar RLS automaticamente

**Estado "inativo" em vez de soft delete:**
- Modelo já tem `ativo` boolean — usar pra "desativar" (esconde da lista por padrão, não aparece em selects de pacotes futuros)
- DELETE físico permitido **apenas** se não houver pacotes vinculados (FK validation)
- Endpoint `DELETE` faz check + retorna 409 se em uso

**Esta story:**
1. Helper `requireAdmin()` análogo ao `requireSuperAdmin()` da 2.1 (em `src/lib/api/admin-guard.ts`)
2. Validators Zod `setorCreateSchema` + `setorUpdateSchema` + `setorListQuerySchema`
3. DB helpers em `src/lib/db/setor.ts` usando `withTenant()` (RLS automático)
4. 5 endpoints REST: GET list, GET detail, POST, PATCH, DELETE
5. UI substitui stub `/admin/setores` — usa Table + Sheet (mesmo pattern da 2.1)
6. **Reusa:** EmptyState, Sheet, Input, Label, Table, Badge, handleApiError, errors.ts (todos da 2.1+2.7)

**Out of scope:**
- Validação de lotação em tempo real (capacidade vs. count atual de pacotes) — futuro
- Reordenação manual (drag-and-drop) — futuro
- Bulk operations — futuro
- Histórico de mudanças (audit_log) — story 8.3
- Importação CSV — story dedicada futura
- Soft delete via `deleted_at` — não está no schema, decisão consciente (usa `ativo`)

## Notas para o @dev

### Decisões pré-aprovadas

- **`withTenant()` em TODAS as queries DB** — RLS aplica filtro por `app.current_condominio` automaticamente. Sem isso, query bypassa isolamento (CRÍTICO). Test de tenant isolation valida.
- **`ativo: boolean` em vez de `deleted_at`** — schema já tem, mais simples pro caso de uso (pode reativar a qualquer momento).
- **DELETE físico apenas se 0 pacotes** — preserva integridade FK histórica. UI prefere "desativar".
- **`requireAdmin()` SÓ admin** — super_admin não passa (story 8.2 vai criar `requireRole(['admin', 'super_admin'])` se necessário).
- **Substituir o stub da 2.7** — não criar caminho paralelo. Se 2.7 não tiver mergeado ainda, vai estar disponível na branch da 2.2 via stacked PR.
- **`<SetorForm>`** — copiar pattern do `CondominioForm` (useState + Zod safeParse, sem RHF). Form simples (4 campos).

### Pontos de atenção

1. **Test de tenant isolation é CRÍTICO.** Cenário: cria setor A no condomínio X via SUPERUSER (bypass RLS), depois faz `listSetores()` dentro de `withTenant(condominio Y)` → deve retornar 0 setores de A. Sem esse teste, qualquer regressão futura no `withTenant` passa silenciosa.

2. **`withTenant()` lança `NoCondominioAssignedError`** se `ctx.condominioId` falta. Não reinventar guard — só chamar.

3. **`_count.pacotes` na lista pode ser 0 hoje** (story 3.x cria pacotes). OK — sempre 0 nesta story, mas estrutura pronta.

4. **Cascade do FK `setor.condominio_id`** já é `CASCADE` no schema. Quando condomínio é arquivado/deletado (story 2.1), setores somem junto (atualmente não acontece pois 2.1 faz soft delete, não DELETE físico).

5. **Endpoint DELETE retorna 409 com mensagem PT-BR explicando** — UX importante (admin precisa entender por que falhou).

6. **Página `/admin/setores`** server component. Layout `(admin)` da 2.7 já garante guard e tenant context. Página apenas chama helper DB.

## Dev Agent Record (2026-05-06)

**Implementação:** @dev (Dex)

### Decisões e desvios

1. **`requireAdmin()` retorna interface explícita** (não `Extract<>`) — `Extract` resolveu pra `never` em runtime/typecheck (TS pegou `role: 'admin' | 'porteiro'` como discriminated não-narrow). Solução: construir objeto retornado manualmente com tipo concreto.
2. **Validator descricao com `z.preprocess`** — `.optional().or(z.literal('').transform(...))` não funcionava (string vazia caía no path string-com-trim). `preprocess` antes do schema converte `''` → `undefined` cleanly.
3. **Helpers DB usam `withTenant()`** (que internamente chama `getTenantContext()`) — caller (route handler) já chamou `requireAdmin()` antes, então auth está garantida. Sem duplicação.
4. **`condominio_id` injetado no `createSetor(data, condominioId)`** — payload Zod NÃO tem o campo (defesa contra escape de tenant via input). Passa explicitamente do `ctx.condominioId` da rota.
5. **DELETE valida `_count.pacotes` no handler** — não no helper DB. Helper é puro, lógica de negócio fica na rota. Mensagem PT-BR explicativa.
6. **Página `/admin/setores`** — pageSize 100 sem paginação visual ainda (admin típico tem <50 setores). Quando virar dor, adicionar paginador igual ao da 2.1.

### Test crítico de tenant isolation

`tests/integration/setor-api.test.ts:96-117` cria setores em condomínio A via `withTenantContext(ctxA)`, depois faz `findMany` dentro de `withTenantContext(ctxB)` → assert que retorna **0 resultados** (RLS isolando corretamente). Sem esse teste, qualquer regressão futura no `withTenant` passaria silenciosa.
