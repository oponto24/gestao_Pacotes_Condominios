## User Story

**Como** admin do cond,
**eu quero** cadastrar funcionários porteiros,
**para que** cada turno de portaria tenha login próprio e os pacotes registrem corretamente quem recebeu/entregou (auditoria FR-019/FR-020/FR-045).

---

## Acceptance Criteria

### AC1 — Página `/admin/funcionarios`
- [ ] Criar `src/app/(admin)/admin/funcionarios/page.tsx`
- [ ] Lista users role=porteiro do cond
- [ ] Mostra: email, nome, status, último login (se Clerk preencheu)
- [ ] Item "Funcionários" na sidebar admin

### AC2 — Form adicionar porteiro
- [ ] `src/components/admin/PorteiroForm.tsx`
- [ ] Inputs: email + nome
- [ ] Submit → POST `/api/admin/users` (mesma API da 8.6, mas com `role=porteiro` no body)

### AC3 — API: aceitar role no body
- [ ] Atualizar `POST /api/admin/users` (criada na 8.6) pra aceitar `role: 'admin' | 'porteiro'` no body
- [ ] Validação: admin pode criar admin OU porteiro do PRÓPRIO cond. Não pode criar super_admin
- [ ] Audit log: `acao='admin_created_by_admin'` ou `acao='porteiro_created_by_admin'`

### AC4 — Reconciliação Clerk (igual 8.5/8.6)
- [ ] Test manual: admin cadastra porteiro com email X → porteiro cria conta Clerk com email X → faz login → entra direto no `/chegada` (role=porteiro, cond do admin)

### AC5 — Tests
- [ ] Adicionar 3 casos no test integration `admin-create-admin.test.ts` (renomear pra `admin-create-user.test.ts`):
  - Admin cria porteiro → user pendente role=porteiro
  - Admin tenta criar super_admin → 400
  - Porteiro do cond NÃO acessa /admin/funcionarios → redirect/403

### AC6 — Quality gates
- [ ] Lint, typecheck, test, build OK
- [ ] Smoke E2E manual:
  - **S1:** Admin cadastra porteiro com email real seu → você cria conta no Clerk com aquele email em outro browser/sessão → consegue entrar como porteiro do cond
  - **S2:** Você cadastrado como porteiro acessa /chegada → tira foto → o `funcionario_recebedor_id` registrado é o user porteiro recém-criado (ver no banco ou em /admin/pacotes/[id])
