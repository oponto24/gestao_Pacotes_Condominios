## User Story

**Como** porteiro,
**eu quero** classificar o tamanho do pacote (Pequeno/Médio/Grande/Extra Grande) e definir onde ele está armazenado (setor + posição),
**para que** o morador receba na notificação WhatsApp (Epic 4) onde retirar e o admin saiba localizar (FR-016, FR-017, FR-018, FR-030).

---

## Contexto

Após `/chegada/confirmar/[id]` (3.8), porteiro chega aqui. Esta é a **última etapa antes da notificação automática**: ao confirmar tamanho/setor/posição, o pacote vira `aguardando_retirada` (FR-030) e a 4.x dispara o WhatsApp.

**Decisão de produto:**
- **Tamanho** é obrigatório (4 opções fixas — TamanhoPacote enum)
- **Setor** é obrigatório (lista vinda do tenant)
- **Posição** é opcional, texto livre ("prateleira 3", "B-12") — FR-018
- **Sugestão de setor por tamanho** é stretch goal — MVP usa primeiro setor ativo como default; lógica configurável por condomínio fica como tech debt registrado

**Fluxo:**
1. GET `/chegada/organizar/{id}` server component
2. Carrega pacote (já confirmado pela 3.8) + setores ativos do tenant
3. Renderiza form com 4 botões grandes de tamanho + select de setor + input posição
4. Submit → PATCH `/api/pacotes/{id}/organizar`
5. Server atualiza pacote: `tamanho`, `setor_id`, `posicao`, `status='aguardando_retirada'`
6. Cria evento `pacote_evento` tipo... (a 4.x cria `notificado` separadamente; aqui usar `confirmado` já foi consumido pela 3.8 — **decisão:** usar metadata expandida em **novo evento implícito** ou aceitar que matching desta etapa só vira mudança de status sem evento dedicado. Vou propor: **registrar como sub-evento adicional** com `tipo='confirmado'` mas com flag `metadata.organizado=true` — não, melhor: **sem novo evento aqui**, status muda e a 4.x cria `notificado`. Histórico fica claro no audit.)
7. Redirect pra `/chegada` (volta pro start) ou `/chegada/sucesso/{id}` (tela de feedback)

**Decisão final:** após PATCH organizar, redireciona pra `/chegada` (ciclo recomeça). Toast de sucesso opcional. Status `aguardando_retirada` sinaliza que está pronto pra notificar. Job de notificação Epic 4 vai detectar.

**Fora do escopo:**
- Sugestão de setor inteligente (tech debt)
- Tela de "pacote registrado com sucesso" (decisão: redirect direto)
- Disparo de WhatsApp (Epic 4)

## Dev Notes

### Pattern reusado
- Server page + lib helper + Zod validator + PATCH endpoint (idêntico à 3.8)
- shadcn Select + Input + Button
- Idempotência via verificação de estado atual

### Nova superfície
- **Primeira mudança de status para `aguardando_retirada`** — sinaliza pra Epic 4 disparar WhatsApp
- **4 botões grandes de tamanho** — pattern visual touch-friendly que pode ser reusado

### Riscos

1. **Sem setores cadastrados:** porteiro não consegue prosseguir. Mitigação: tela mostra mensagem amigável; CRUD admin de setores entra na 6.5 ou tech debt. Para piloto, criar setor "Recepção" via seed.
2. **Race condition:** porteiro abre 2 abas com mesmo pacote. Idempotência mitiga.
3. **Tamanho subjetivo:** "Médio" varia entre porteiros. Tech debt: tooltip com exemplos visuais.

### Sugestões para Dev Notes

1. **Bottom sheet em mobile** para o select de setor pode ter UX melhor que dropdown padrão. Tech debt se shadcn Select não comportar.
2. **`aria-live="polite"`** nos botões de tamanho pra leitor de tela anunciar seleção.
3. **Persistir tamanho selecionado em localStorage** — se porteiro recarregar, mantém estado. Tech debt minor.
