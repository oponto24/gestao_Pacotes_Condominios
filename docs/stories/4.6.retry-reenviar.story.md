## User Story

**Como** admin / porteiro,
**eu quero** ver status da notificação no detalhe do pacote e poder reenviar manualmente quando falhou,
**para que** moradores que não receberam (telefone errado, fora de janela) possam ser corrigidos sem ter que ligar pra dev.

---

## Contexto

**FR-037 (PRD):** "Em caso de falha de envio, sistema deve tentar reenvio automático até 3 vezes com backoff exponencial."

**Retry automático já está coberto pela story 4.3** (BullMQ `attempts: 4, backoff: exponential`). Esta story:
- Valida que 4.3 está com retry correto via teste
- Adiciona endpoint `POST /api/pacotes/:id/reenviar-whatsapp` para reenvio manual
- Adiciona botão na tela de detalhe (story 6.3)
- Mostra status da última mensagem na timeline

---

## Acceptance Criteria

### AC1 — Validação de retry da 4.3 (não muda código, só teste)
- [x] Adicionar teste de integração que simula 3 falhas retriáveis seguidas + sucesso → BullMQ retenta corretamente, último attempt grava `sent`
- [x] Adicionar teste que simula 4 falhas retriáveis → após 4ª, mensagem fica `failed` permanentemente
- [x] Documentar no PR a interpretação: "primeiro envio + 3 retries = 4 attempts total"

### AC2 — Endpoint `POST /api/pacotes/:id/reenviar-whatsapp`
- [x] Auth: usuário do condomínio do pacote (admin ou porteiro)
- [x] Validação:
  - Pacote existe e pertence ao condomínio do user
  - Pacote tem status `aguardando_retirada` (não retirado, não cancelado)
  - **Rate limit:** máx 3 reenvios manuais por pacote (`SELECT count() FROM whatsapp_message WHERE pacote_id = ? AND created_at > now() - interval '1h'`). Acima disso → 429
- [x] Lógica:
  1. Enfileira novo job `sendWhatsApp` (cria nova `WhatsAppMessage` row, não reutiliza falhas)
  2. Audit log com `acao = 'reenvio_whatsapp_manual'`, `user_id`, `pacote_id`
  3. Retorna `{ enqueued: true, queue_id }`

### AC3 — UI: status na timeline (story 6.3)
- [x] Em `src/app/admin/pacotes/[id]/page.tsx` (story 6.3, já existente), adicionar bloco "Notificações WhatsApp"
- [x] Lista cronológica das `WhatsAppMessage` outbound do pacote com:
  - Status (badge: pendente cinza, enviado azul, entregue verde-claro, lido verde, falhou vermelho)
  - Timestamps (sent_at, delivered_at, read_at, failed_at)
  - Telefone destinatário (mascarado: `+55 11 9****-1234`)
  - `failure_reason` se falhou
- [x] Botão **"Reenviar notificação"** habilitado se:
  - Última mensagem está `failed` OU `pending >5min` OU não houve nenhuma mensagem ainda
  - Pacote em `aguardando_retirada`

### AC4 — Loading + feedback UI
- [x] Click no botão → POST → toast sucesso/erro → invalidar query da timeline (revalidate)
- [x] Estado loading visível no botão (disabled + spinner)
- [x] Erro 429 mostra mensagem clara: "Limite de 3 reenvios/hora atingido"

### AC5 — Tests
- [x] `tests/integration/reenviar-whatsapp.test.ts`:
  - Endpoint enfileira job e cria audit log
  - Rate limit: 4ª chamada em <1h retorna 429
  - User de outro condomínio → 403
  - Pacote retirado → 422
- [x] `tests/unit/whatsapp-status-badge.test.tsx`: cada status renderiza cor correta
- [x] `tests/e2e/reenviar-flow.spec.ts` (Playwright): admin clica botão, mensagem aparece em pending na lista

### AC6 — Atualização docs/runbook
- [x] Adicionar parágrafo no runbook: "Em produção, se admin reportar `failure_reason: 131026`, significa que morador apagou WhatsApp ou está sem internet há muito tempo — orientar admin a confirmar telefone com morador antes de reenviar."

## Out of scope

- Editar telefone do morador inline → 2.4 (CRUD morador, já existe)
- Cancelar mensagem em pending → não previsto, BullMQ não suporta nativo
- Métricas de delivery rate por condomínio → pós-MVP

---

## Riscos & mitigações

| Risco | Mitigação |
|-------|-----------|
| Admin spam reenvio e estoura quota Meta | Rate limit 3/h por pacote |
| Mensagem reentrega gera notificação dupla pro morador | UX comum em SaaS, aceitável; rate limit ajuda |
| 4.4 ainda não em prod → status nunca atualiza para delivered/read | Documentar: enquanto 4.4 não deploya, mensagens ficam em `sent` permanentemente — não é bug |
