## User Story

**Como** sistema,
**eu quero** que o worker pegue o job `sendWhatsApp` enfileirado quando porteiro organiza um pacote (story 3.9),
gere o QR (4.2), envie o template `pacote_chegou` via Meta (4.1) e registre `WhatsAppMessage` no banco,
**para que** o morador receba notificação automática segundos após a portaria organizar o pacote.

---

## Contexto

**FR-030 (PRD):** "Após o pacote ser registrado e organizado (setor + posição definidos), sistema deve disparar notificação WhatsApp automaticamente."
**FR-037:** "Em caso de falha de envio, sistema deve tentar reenvio automático até 3 vezes com backoff exponencial."
**NFR-003:** "Tempo de envio de notificação WhatsApp: <10 segundos após confirmação do setor (P95)."

**Esta story conecta os pedaços.** 4.1 fez o cliente Meta, 4.2 o QR. Agora orquestramos.

**Fluxo do job:**
1. Trigger: ao final de 3.9 (organizar), enfileira `sendWhatsApp` com `{ pacote_id }`
2. Worker carrega pacote + condomínio + morador alvo + WhatsAppNumber compartilhado
3. Determina destinatário via 4.5 (que vem depois — por enquanto: morador da unidade, fallback condômino principal)
4. Garante QR via `ensureQrForPacote` (4.2) → URL pública
5. Cria `WhatsAppMessage` em estado `pending`
6. Chama `sendTemplate` (4.1) com template `pacote_chegou`, headerImage = QR URL, bodyParams = [nome, condomínio]
7. Atualiza `WhatsAppMessage`: `meta_message_id = wamid`, `status = sent`, `sent_at = now()`
8. Em erro: `status = failed`, `failure_reason`, `failed_at`, `retry_count++`. BullMQ retry automático cuida do reenvio.

## Acceptance Criteria

### AC1 — Job processor `src/lib/queue/jobs/send-whatsapp.ts`
- [ ] Constante `SEND_WHATSAPP_JOB_NAME = 'sendWhatsApp'`
- [ ] Payload: `{ pacote_id: string; condominio_id: string }`
- [ ] BullMQ retry config: `attempts: 4`, `backoff: { type: 'exponential', delay: 5_000 }` (5s, 10s, 20s, 40s)

### AC2 — Lookup do número WhatsApp compartilhado
- [ ] Query: `WhatsAppNumber` com `condominio_id IS NULL AND ativo = true` (modelo CON-005)
- [ ] Se não houver registro, lançar erro fatal (operação SaaS está mal configurada; não tem o que tentar de novo)

### AC3 — Determinação do destinatário (versão simplificada — substituída em 4.5)
- [ ] Por agora: pega `condominio_principal_morador_id` da unidade do pacote
- [ ] Se ausente → registro `WhatsAppMessage` com `status = failed`, `failure_reason: 'sem_destinatario'`, **NÃO** retry (fatal)
- [ ] **TODO comment:** `// 4.5 vai substituir esta lógica por matching nome → adicionais → fallback`

### AC4 — Geração do QR
- [ ] Chama `ensureQrForPacote(pacoteId)` (story 4.2) — idempotente
- [ ] Pega `publicUrl` retornado pra usar como `headerImageUrl`

### AC5 — Criação `WhatsAppMessage` em transação
- [ ] Insert antes do envio (`status: pending`) com:
  - `condominio_id`, `whatsapp_number_id` (id do registro buscado em AC2)
  - `direction: outbound`
  - `to_phone`, `from_phone` (display_phone do WhatsAppNumber)
  - `template_name: 'pacote_chegou'`
  - `template_params: { body: [nome, condominio], header: { type: 'image', url: qrUrl } }`
  - `pacote_id`, `morador_id`
- [ ] Após chamar Meta, update mesma row: `meta_message_id`, `status: sent`, `sent_at`
- [ ] Em falha: update com `status: failed`, `failure_reason`, `failed_at`, `retry_count++`

### AC6 — Tratamento de erros (`MetaApiError` da 4.1)
- [ ] `error.retriable === true` → throw para BullMQ retry
- [ ] `error.retriable === false` → marca `failed`, **NÃO** throw (BullMQ não retenta)
- [ ] Log estruturado com `pacote_id`, `meta_code`, `fbtrace_id`

## Riscos & mitigações

| Risco | Mitigação |
|-------|-----------|
| Worker tenta enviar antes de Meta aprovar template | `META_DISABLED=true` em CI; smoke manual só com aprovação |
| URL pública do QR não acessível pela Meta em dev local | Smoke real exige tunnel (ngrok) ou staging |
| Race condition: dois jobs pra mesmo pacote | `WhatsAppMessage` não tem unique por `pacote_id` (pode haver retry/reenviar manual). OK no MVP. |
