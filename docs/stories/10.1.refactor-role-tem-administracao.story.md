## User Story

**Como** sistema,
**eu quero** suportar dois níveis de administrador (`admin_master` e `admin_funcionario`) e uma flag `tem_administracao` no condomínio,
**para que** o Epic 10 possa bifurcar o fluxo de organização do pacote (porteiro → admin) em condomínios com equipe administrativa dedicada, sem quebrar nenhuma funcionalidade existente.

---

## Contexto

**PRD §3.8 (FR-080 a FR-085):** define hierarquia operacional com 4 roles (`super_admin`, `admin_master`, `admin_funcionario`, `porteiro`) + flag `condominio.tem_administracao` que altera o fluxo de chegada do pacote.

**Esta story é puramente refactor + foundation:** nenhuma mudança visível pro usuário final. Funcionalmente preserva o comportamento atual (admin existente vira `admin_master`, sem `admin_funcionario` ainda em uso). É a base que stories 10.2-10.7 vão estender.

### Decisões importantes

- **Migration data idempotente:** `UPDATE user SET role='admin_master' WHERE role='admin'` antes do enum ser alterado. Prisma migration faz isso em ordem (data migration → schema migration).
- **Sem renomear coluna:** o enum continua chamado `UserRole`. Só os valores mudam.
- **Default da flag:** `tem_administracao=false` pra preservar comportamento atual. Story 10.3 adiciona toggle UI; super-admin terá que ativar manualmente nos condomínios que têm equipe.
- **Guards com 4 níveis de granularidade** (ver AC4) cobrem casos do Epic 10 sem refactor adicional.

- [ ] `npx prisma generate` roda sem erros
- [ ] `npx prisma format` mantém schema válido

### AC2 — Migration de dados + schema

- [ ] Criar `prisma/migrations/{timestamp}_role_hierarquia_operacional/migration.sql` com:
  ```sql
  -- 1. Adicionar tem_administracao
  ALTER TABLE "condominio" ADD COLUMN "tem_administracao" BOOLEAN NOT NULL DEFAULT false;

  -- 2. Renomear enum value (Postgres: ALTER TYPE ... RENAME VALUE)
  ALTER TYPE "UserRole" RENAME VALUE 'admin' TO 'admin_master';

  -- 3. Adicionar novo valor admin_funcionario
  ALTER TYPE "UserRole" ADD VALUE 'admin_funcionario';
  ```
- [ ] Migration aplica localmente (Postgres docker compose) sem erro
- [ ] Migration aplica em prod via `prisma migrate deploy` (smoke pós-merge)
- [ ] Verificar que usuários existentes ficam com `role='admin_master'`

## Riscos & mitigações

| Risco | Probabilidade | Impacto | Mitigação |
|-------|---------------|---------|-----------|
| Migration `ALTER TYPE ... RENAME VALUE` quebra em prod | Baixa | Alto | Postgres 13+ suporta. Já estamos em 16. Smoke local antes de deploy |
| Caller missed (algum `requireAdmin` esquecido) | Média | Médio | `grep` final cobrindo todos os callers + typecheck (deprecated alias garante que mesmo se faltar, funciona) |
| Tests existentes assumem string literal `'admin'` em fixtures | Média | Baixo | `grep -rn "'admin'" tests/` revela todos os casos pra atualizar |
| `prisma migrate deploy` em prod falha por ordem de migrations | Baixa | Alto | Migration é forward-only (sem rollback automático no enum). Backup do banco antes do deploy |
