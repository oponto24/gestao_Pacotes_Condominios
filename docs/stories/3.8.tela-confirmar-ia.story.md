## User Story

**Como** porteiro,
**eu quero** revisar e ajustar os dados que a IA extraiu da etiqueta antes de confirmar o pacote,
**para que** mesmo quando a IA errar (FR-014 modo "sempre confirmar"), o cadastro saia certo — destinatário correto vinculado à unidade certa.

---

## Contexto

A 3.6 redireciona pra `/chegada/confirmar/{pacote_id}` após criar o rascunho. A 3.5 + 3.7 já preencheram (no melhor caso): `ia_extracao_raw`, campos textuais (`nome_destinatario_etiqueta`, `cep_etiqueta`, etc.), e — se matching deu certo — `unidade_id` + `destinatario_id`.

**FR-014:** Sempre confirmar antes de gravar. Mesmo com confiança alta, porteiro deve revisar.

**Esta story entrega:**
1. Página server-side `/chegada/confirmar/[id]` que carrega pacote + match + foto
2. Form client `IAExtractionForm` editável (5 campos textuais + unidade + destinatário)
3. Endpoint `PATCH /api/pacotes/{id}/confirmar` que valida + grava + muda status
4. Pipeline de busca de unidade/morador inline (se IA não casou ou user quer trocar)

**Status do pacote após confirmar:** vira `aguardando_retirada` se for caso "matched" + dados completos. Story 3.9 organiza tamanho/setor antes da notificação WhatsApp (4.x). **Decisão:** o status `aguardando_retirada` só é setado **depois** da 3.9 (organizar). Aqui na 3.8 marcamos `confirmado` (criar enum value novo).

**Espera aí:** olhando schema — `PacoteStatus` tem `rascunho | pendente_identificacao | aguardando_retirada | retirado | cancelado`. Não há `confirmado` separado. **Decisão revisada:** após 3.8, status fica `rascunho` mas com flag implícita "pronto para 3.9" (presença de `unidade_id` + `destinatario_id` + `confirmed_em` timestamp). 3.9 promove para `aguardando_retirada` ao definir tamanho/setor.

Adicionar campo `confirmed_em` ao schema (migração pequena) ou usar `pacote_evento` tipo `confirmado` (já existe!) como sinalização. **Decisão final:** usar `pacote_evento` tipo `confirmado` (sem migração de schema), e front-end consulta evento mais recente para saber se já foi confirmado.

**Fluxo:**
1. GET `/chegada/confirmar/{id}` (server component)
2. Carrega pacote + foto principal + dados extraídos + match + unidade.identificador + morador.nome
3. Renderiza `<IAExtractionForm>` com valores pré-preenchidos
4. Porteiro edita campos se necessário; pode buscar outra unidade/morador via combobox
5. Submit → PATCH `/api/pacotes/{id}/confirmar` com payload editado
6. API valida (Zod) + atualiza pacote + cria evento `confirmado` + redireciona pra `/chegada/organizar/{id}` (3.9)
7. Se 3.9 ainda não existe (404) — tudo bem, smoke valida que confirmação salvou no DB

## Dev Notes

### Pattern reusado
- `requirePorteiro()` guard (3.4)
- `withTenant` para queries com RLS (2.x)
- Zod validators (3.4, 3.6)
- `pacote_evento` para audit trail (3.4, 3.5, 3.7)

### Nova superfície
- **Primeira página server-side com lookup tenant-scoped via parametro de URL**
- **Combobox shadcn** — pode ser primeira vez no projeto
- **Endpoint PATCH idempotente** com idempotência via evento existente

### Decisões importantes

1. **Sem novo enum status:** evento `confirmado` (já existe em `PacoteEventoTipo`) sinaliza confirmação. Status muda só na 3.9 → `aguardando_retirada`.
2. **Unidade obrigatória no submit:** mesmo que IA tenha errado, porteiro precisa selecionar uma. Se não tem unidade cadastrada, admin precisa cadastrar antes (volta pra 2.2).
3. **"Outra pessoa" como destinatário:** sem campo novo no schema. Se admin escolher "outra pessoa", `destinatario_id` fica `null`, `destinatario_resolvido_via='manual_override'`, e o nome digitado vai pra `nome_destinatario_etiqueta` (semântica passa a ser "nome final do destinatário"). IA original preservada em `ia_extracao_raw` para auditoria.
4. **Sem upload de foto adicional:** foto é só a inicial. Edição de foto fica como tech debt.
5. **Auto-foco quando confiança baixa:** se `ia_confianca < 0.5`, foco inicial no campo Nome (revisão prioritária).

### Riscos

1. **Schema gap:** verificar se `pacote.destinatario_outro_nome` ou similar existe; senão, adicionar campo via migração pequena (ou usar `pacote_evento.metadata` como tech debt).
2. **N+1 queries:** loading inicial pode disparar 4-5 queries (pacote, foto, unidade, morador, eventos). Considerar `select` com includes — Prisma já otimiza, mas medir.
3. **Combobox em mobile:** componentes shadcn `<Command>` podem ter UX ruim em telas pequenas. Fallback: `<select native>` em viewport <640px.
4. **Race condition:** 2 abas abertas confirmando ao mesmo tempo → mitigado pela idempotência (segunda dispara `already_confirmed`).
