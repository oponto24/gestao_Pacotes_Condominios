# Story 13.6 — Error messages genéricas + impersonate cookie hardening

**Epic:** E13 — Hardening UI/UX + Segurança Frontend (P0)
**Status:** Done
**Owner:** @dev (Dex)
**Tipo:** Security hardening
**Severidade:** BAIXA
**Estimativa:** 0.5 dia

---

## User Story

**Como** sistema,
**eu quero** que mensagens de erro não revelem estrutura interna e que cookies de impersonate sejam invalidados corretamente,
**para que** atacantes não consigam inferir schema do banco ou manter sessões fantasma.

---

## Acceptance Criteria

### AC1 — Error messages genéricas no client
- [x] Componentes admin que fazem `alert(body.message)` substituídos por toast genérico
- [x] Em produção: "Operação falhou. Tente novamente."
- [x] Detalhes técnicos apenas em `console.error` (não no UI)

### AC2 — Zod validation sem field names
- [x] `handle-error.ts` retorna `{ ok: false, code: 'validation_error' }` sem `fields` detalhados em produção
- [x] Em dev (`NODE_ENV=development`): mantém `fields` para debugging

### AC3 — Impersonate cookie invalidação
- [x] Middleware verifica `maxAge` do cookie de impersonate server-side
- [x] Se cookie expirado (>8h desde criação): deleta automaticamente e redireciona
- [x] `/api/super-admin/impersonate/stop` limpa cookie e invalida em audit log

### AC4 — Testes
- [x] Test: error handler não retorna fields em production
- [x] Test: impersonate stop limpa cookie
