## User Story

**Como** porteiro,
**eu quero** bipar o código de barras (1D ou QR) do pacote com a câmera do
celular OU digitar manualmente se o bipe falhar,
**para que** o sistema tenha rastreio único do pacote sem depender só da foto.

---

## Contexto

**FR-011 (PRD):** "Sistema deve permitir bipar o código de barras do pacote
(1D ou QR) usando a câmera do celular — opcional, serve para unicidade e
rastreio."
**UX_SPEC §4.1:** "Bipe ativo: modal full-screen com scanner html5-qrcode,
botão fechar (X)."
**UX_SPEC §3.3 organisms:** "BarcodeScannerInput (câmera + input fallback)"

**Esta story entrega APENAS o componente reusável `<BarcodeScannerInput>`.**
Não implementa a tela `/chegada` completa nem chama API — isso é story 3.6.
Diferença com 3.2: bipe é **opcional** (FR-011), foto é obrigatória (FR-012).

**API esperada:**
```tsx
<BarcodeScannerInput
  value={codigo}
  onChange={(codigo) => setCodigo(codigo)}
  onError={(msg) => /* opcional */}
  placeholder="Código do pacote (opcional)"
/>
```

Retorna **string única** (código lido). Composto: input texto editável + botão "Bipar" que abre modal fullscreen com scanner.

## Notas para o @dev

### Decisões pré-aprovadas

1. **`html5-qrcode` direto** (não BarcodeDetector API): BarcodeDetector ainda tem suporte limitado em iOS Safari (até 2025) e nem todos browsers expõem. html5-qrcode é polyfill confiável + ~50KB. Decisão pragmática para MVP.
2. **Modal fullscreen** (não inline): scanner precisa de viewport grande pra detecção confiável de códigos pequenos em etiquetas. Inline fica apertado.
3. **Detecção fecha modal automaticamente:** UX rápida — porteiro não precisa confirmar. Se ele bipou, é porque era o que queria. Pode editar manualmente depois se errado.
4. **Input texto SEMPRE editável** (mesmo após bipe): porteiro pode corrigir/sobrescrever. Não trava em "auto-only".
5. **Formatos suportados:** QR_CODE + CODE_128 + CODE_39 + EAN_13 + EAN_8 + UPC_A. Cobertura de Correios (CODE_128 dominante), e-commerce BR (CODE_128 + EAN), QR (futuro). Excluímos formatos exóticos (CODABAR, ITF) pra reduzir falsos positivos.
6. **Reutiliza `<Sheet>` da 2.4**: já existe componente de drawer fullscreen no projeto. Só usar `side="bottom"` ou full sheet pra UX mobile.
7. **`useReducer` para state machine:** mesmo pattern da 3.2 (consistência) — 4 estados: closed/requesting/scanning/error.
8. **Vibration ao detectar:** mesma microinteraction da 3.2 (50ms).

### Pontos de atenção

1. **HTTPS obrigatório em produção:** mesmo da 3.2 (getUserMedia interno do html5-qrcode).
2. **Cleanup html5-qrcode é EM 2 PASSOS:** `instance.stop()` (para o stream) + `instance.clear()` (limpa o DOM do div). Esquecer um deles vaza memória.
3. **`Html5Qrcode` tem promise-based API:** `start()` retorna Promise — wrap em try/catch + state machine.
4. **iOS Safari pode falhar silenciosamente:** se câmera não inicializa, html5-qrcode às vezes não rejeita a Promise. Adicionar timeout (5s) que dispara erro se scanner não emitir nada.
5. **Detecção pode disparar múltiplas vezes:** debounce ou flag "already detected" pra ignorar callbacks subsequentes do mesmo código.
6. **`formatsToSupport` é array de enum:** importar `Html5QrcodeSupportedFormats` e usar valores explícitos.
7. **Container precisa estar montado quando `start()` chamado:** usar `useRef` + `useEffect` que aguarda mount. Tipicamente faz sense fazer `start()` em `useEffect` que observa `state.kind === 'requesting'`.

### Não fazer agora

- API POST /api/pacotes (envio do código) → 3.4
- Detecção de duplicatas (código já existe no banco) → 3.4 ou 3.10 (validação no backend)
- Suporte a múltiplos códigos no mesmo scan → fora MVP
- Histórico de códigos bipados → fora MVP
- BarcodeDetector API (futuro) → migrar pós-MVP se quiser tirar dep externa
- Customização de cores do scanner overlay → fora MVP

## Dev Notes

### Pattern reusado de stories anteriores
- State machine via `useReducer` — pattern PhotoCapture 3.2
- Cleanup determinístico em useEffect — pattern PhotoCapture 3.2
- `<Sheet>` componente — story 2.4 (Sheet do MoradorForm)
- `cn()` helper — utility padrão
- Vibration API — pattern PhotoCapture 3.2

### Nova superfície
- **`html5-qrcode`** — primeira lib de scanning. Adiciona ~50KB ao bundle.
- **`src/components/portaria/`** continua crescendo (3 componentes: PortariaLayout, PhotoCapture, BarcodeScannerInput).
- **Modal fullscreen** com Sheet — primeiro uso bottom/fullscreen no projeto.

### Riscos

1. **Bundle size:** html5-qrcode (~50KB) + lucide icons + componente. Bundle de portaria já tinha 11.6KB com 3.2; com 3.3 vai pra ~70KB+ provavelmente. Aceitável; monitorar se passar de 200KB.
2. **Detecção falha em códigos baixa qualidade:** etiquetas amassadas/borradas podem não bipar. Fallback de digitação manual mitiga.
3. **iOS Safari + Sheet bottom:** keyboard pode esconder input quando aberto. Testar em mobile real.
4. **Race condition entre `start()` e `unmount`:** se user fecha modal rápido após abrir, `start()` ainda em curso. Cleanup com flag `cancelled` (mesmo pattern 3.2).
5. **Múltiplos formatos podem causar lentidão:** quanto mais formatos `formatsToSupport`, mais lento. 6 formatos é equilibrado.
