## User Story

**Como** sistema,
**eu quero** casar automaticamente os dados extraídos pela IA com a unidade e morador certos do condomínio,
**para que** o porteiro só revise (não digite) — e quando o casamento falhar, o pacote vire **pendente_identificacao** pra resolução manual (FR-021).

---

## Contexto

A 3.5 deixa o pacote com `ia_extracao_raw` preenchido (nome, CEP, complemento, transportadora). Falta **transformar essa extração em vínculos do banco** (`pacote.unidade_id`, `pacote.destinatario_id`, `pacote.destinatario_resolvido_via`).

**FR-015:** Sistema deve sugerir automaticamente o casamento com a unidade do condomínio baseado em CEP + nome + complemento.
**FR-021:** Se o casamento automático falhar, pacote cai em **pendente_identificacao** para resolução manual pelo admin.
**FR-031/FR-032:** Notificação prioriza nome do destinatário cadastrado; senão fallback pro condômino principal.

**Esta story entrega APENAS a lógica + integração no worker.** A UI de confirmação fica na 3.8. A tela de pendentes fica na 3.10.

**Algoritmo (3 estágios):**

1. **Gate CEP:** se `cep_etiqueta` existe e não bate com `condominio.cep`, marca `outside_cep` (não bloqueante; só baixa confiança).
2. **Match unidade** via complemento normalizado:
   - Extrai número de apto/bloco do `complemento_etiqueta` ("AP 1304 bloco 3" → `{apto:'1304', bloco:'3'}`)
   - Busca `Unidade` onde `identificador` contém o número e (se bloco extraído) `bloco` contém o bloco
   - Empate múltiplo → `ambiguous_unidade`
3. **Match morador** dentro da unidade:
   - Normaliza `nome_destinatario` (lowercase + sem acento) e compara com `morador.nome_normalizado`
   - Trigram similarity (pg `pg_trgm`) ou Levenshtein ≥ 0.7 → match
   - Se achou: `destinatario_resolvido_via = 'destinatario_cadastrado'`
   - Se não achou mas unidade tem `is_principal=true`: `destinatario_resolvido_via = 'fallback_principal'`
   - Se nem isso: `not_found_morador`

**Resultado final:**

| Resultado | `pacote.status` | `unidade_id` | `destinatario_id` | `resolvido_via` |
|---|---|---|---|---|
| Matched (destinatário) | `rascunho` | UUID | UUID | `destinatario_cadastrado` |
| Matched (fallback principal) | `rascunho` | UUID | UUID principal | `fallback_principal` |
| Sem unidade ou ambiguous | `pendente_identificacao` | null | null | null |
| Sem morador na unidade (sem principal) | `pendente_identificacao` | UUID | null | null |

## Acceptance Criteria

### AC1 — Função pura `matchUnidadeMorador`

- [x] Criar `src/lib/matching/match-unidade-morador.ts`
- [x] Assinatura:
  ```ts
  type MatchInput = {
    condominio_cep: string;
    ia: {
      nome_destinatario: string | null;
      cep: string | null;
      complemento: string | null;
    };
  };
  type MatchResult =
    | { kind: 'matched'; unidade_id: string; destinatario_id: string;
        resolvido_via: 'destinatario_cadastrado' | 'fallback_principal'; flags: MatchFlags }
    | { kind: 'pending'; unidade_id: string | null; reason: string; flags: MatchFlags };
  type MatchFlags = { cep_diverge: boolean; complemento_extraido: ParsedComplemento | null };
  ```
- [x] Recebe `tx: Prisma.TransactionClient` para queries (worker tem acesso super-admin)
- [x] Função 100% testável: parsing de complemento, normalização de nome, decisão final

### AC2 — Parser de complemento

- [x] Submódulo `src/lib/matching/parse-complemento.ts` com `parseComplemento(text: string | null) → ParsedComplemento | null`
- [x] Detecta padrões comuns:
  - `"AP 1304"` → `{ apto: '1304' }`
  - `"Apto 301-B"` → `{ apto: '301', bloco: 'B' }`

- [x] Tolerante a abreviações: `apto`, `ap.`, `ap`, `apartamento`, `bl`, `bloco`, `tr`, `torre`
- [x] Retorna `null` se não conseguir extrair número

## Dev Notes

### Pattern reusado
- Worker BullMQ + Prisma direct (sem RLS, super-admin) — igual 3.5
- `pacote_evento` log forense — igual 3.4 e 3.5
- Pure function pattern — igual `captura-page-reducer.ts` da 3.6

### Nova superfície
- **Primeiro algoritmo de domínio puro** — matching é pura lógica (não UI, não rede)
- **Estabelece pattern em `src/lib/matching/`** — pode crescer com matching de telefone (4.x), código ML (7.x)

### Riscos

1. **Falso positivo em matching:** algoritmo retorna unidade errada com confiança alta. Mitigado por:
   - Threshold 0.7 (Levenshtein) — conservador
   - Story 3.8 sempre exige confirmação humana antes de gravar (FR-014)
   - Worker grava `flags` em `pacote_evento.metadata` pra auditoria
2. **Performance com 5k+ moradores:** carregar todos em memória pra scorear é OK até ~10k. Acima disso, migrar pra trigram em SQL (tech debt registrado).
3. **`nome_normalizado` desatualizado:** depende da 2.4 popular esse campo no insert/update. Se vier vazio, fallback pra normalizar on-the-fly.
4. **Complementos exóticos:** "Cobertura", "Loft 5", "Sala 12" podem não casar com regex. Tolerável — vão pra `pendente_identificacao`, admin resolve.

### Sugestões para Dev Notes

1. **Logger estruturado:** `loggerForJob` com `pacote_id` + `result.kind` + `flags` em todo o pipeline. Facilita debug.
2. **Performance benchmark:** medir tempo do matching quando moradores >1000 para validar abordagem in-memory.
3. **Não chamar IA novamente para reanalisar etiqueta** — manter custo zero. Matching é 100% local.
4. **Considerar prefixo "AP"/"APTO" no `Unidade.identificador`:** alguns condomínios cadastram "Apto 101", outros só "101". Normalizar identificador antes de comparar (extrair só dígitos pra match numérico).
