## User Story

**Como** desenvolvedor preocupado com isolamento multi-tenant,
**eu quero** as RLS (Row-Level Security) policies da Dara aplicadas no Postgres + role `webhook_worker` com `BYPASSRLS` criada,
**para que** dados de um condomínio fiquem matematicamente isolados de outros (defesa em profundidade contra bug aplicacional) e o handler de webhook Meta consiga fazer lookup global por telefone (cross-tenant) usando role separada e segura.

---

## Contexto

A Dara entregou o arquivo `prisma/migrations/20260506200212_initial_schema/rls_policies.sql` que:
- Cria 2 helpers SQL: `app_current_condominio()` e `app_is_super_admin()`
- Habilita RLS em 8 tabelas tenant-scoped
- Cria policy `tenant_isolation` em cada uma com `USING (condominio_id = current_setting('app.current_condominio')::uuid OR is_super_admin)`
- Cria role `webhook_worker` com `BYPASSRLS` + grants mínimos (SELECT em morador, INSERT/UPDATE em whatsapp_message e codigo_ml_pendente)

**Atenção crítica do QA da story 1.3:** Prisma **NÃO aplica** `rls_policies.sql` automaticamente — só processa `migration.sql`. Esta story aplica via `psql -f` e cria infra para aplicar idempotentemente nas próximas migrations.

**Out of scope (vem em 1.6):**
- Middleware aplicacional (`getTenantContext()` + `SET LOCAL`)
- Integração com Clerk pra extrair `condominio_id` do user

Esta story só prepara o **banco**. O middleware vem depois.

## Notas para o @dev

### Sobre o método de aplicação do `rls_policies.sql`

Quinn destacou: Prisma NÃO aplica esse arquivo. Você tem 3 opções:

1. **`psql -f` via container exec** (recomendado — mais simples, sem dependência extra):
   ```bash
   docker compose -f infra/docker/docker-compose.yml exec -T postgres \
     psql -U app -d gestao_pacotes < prisma/migrations/20260506200212_initial_schema/rls_policies.sql
   ```
   Note o `-T` (sem TTY) e o `<` (stdin) — funciona porque o psql lê SQL do stdin.

2. **Mount do arquivo via volume e usar `\i`** — mais complexo, precisa mudar compose.

3. **Wrapper Node usando `pg`** — overengineering pra MVP, descartar.

Vá com **opção 1**. O `apply-rls.sh` deve fazer:
```bash
#!/usr/bin/env bash
set -euo pipefail
[[ -z "${WEBHOOK_WORKER_DB_PASSWORD:-}" ]] && { echo "WEBHOOK_WORKER_DB_PASSWORD not set"; exit 1; }
docker compose -f infra/docker/docker-compose.yml exec -T postgres \
  psql -U app -d gestao_pacotes < prisma/migrations/20260506200212_initial_schema/rls_policies.sql
docker compose -f infra/docker/docker-compose.yml exec -T postgres \
  psql -U app -d gestao_pacotes -c "ALTER ROLE webhook_worker WITH PASSWORD '${WEBHOOK_WORKER_DB_PASSWORD}';"
echo "RLS aplicado."
```

### Sobre testes Vitest contra Postgres real

O Vitest atual (story 1.1) usa jsdom — bom pra componentes React, **inadequado pra DB**. Opções:

1. **Vitest com workspace separado** — projeto `unit` (jsdom) + projeto `integration` (node) no mesmo `vitest.config.ts`. Recomendado.
2. **Script separado `test:integration`** que roda `vitest --environment node tests/integration`. Mais simples mas duplica config.

Vá com a **opção mais simples** (2) por enquanto. Quando integration tests crescerem, refatora pra workspace.

Skip automático se Postgres não acessível:
```typescript
import { describe, it, beforeAll, afterAll } from 'vitest';
import { PrismaClient } from '@prisma/client';

const db = new PrismaClient();
let dbAvailable = true;

beforeAll(async () => {
  try { await db.$queryRaw`SELECT 1`; } catch { dbAvailable = false; }
});
afterAll(async () => { await db.$disconnect(); });

describe.skipIf(!dbAvailable)('RLS isolation', () => { /* ... */ });
```

### Decisões pré-aprovadas

- Senha `webhook_worker` é gerada localmente, **nunca commitada**, fica em `.env.local`
- `apply-rls.sh` é idempotente porque o SQL da Dara usa `DROP POLICY IF EXISTS` + `CREATE POLICY` (já é safe-rerun)
- Fixtures de teste usam nomes únicos (ex: `Cond Test 1.4 ${Date.now()}`) pra evitar colisão se rodadas em paralelo
- Não criar pgbouncer ainda — `webhook_worker` conecta direto

### Pontos de atenção

1. **Senha `webhook_worker` é a CHAVE crítica de segurança.** Se vazar, atacante tem acesso BYPASSRLS ao banco. Em produção (story 8.4) virá de Docker secret. Em dev: `.env.local`.

2. **Role precisa ter `LOGIN` explícito.** Postgres por default cria roles sem LOGIN. O SQL da Dara faz `CREATE ROLE webhook_worker WITH LOGIN BYPASSRLS PASSWORD NULL;` — se precisar reaplicar, garantir que LOGIN não foi removido.

3. **Testes RLS precisam estar dentro de transação** ou criar/deletar fixtures sempre. Isolar entre runs.

4. **`SET LOCAL` exige transação ativa.** Em testes Prisma, usar `db.$transaction(async (tx) => { ... })` pra garantir contexto persistente.

5. **NÃO esquecer de validar como `webhook_worker`.** Fácil esquecer e testar tudo como `app` (que ignora RLS porque não tem context setado e a policy permite OR `is_super_admin`). Conectar via `psql -U webhook_worker`.

6. **Cleanup é importante.** Testes que criam fixtures sem limpar viram lixo no banco. Use `afterAll` ou `db.$transaction` com rollback.

## Dev Notes

### Decisões e descobertas críticas

**1. Bug no SQL da Dara: `GRANT CONNECT ON DATABASE CURRENT_DATABASE()`**
Sintaxe inválida — Postgres não aceita função em `GRANT CONNECT`, exige nome literal. Substituído por `GRANT CONNECT ON DATABASE gestao_pacotes`.

**2. RLS owner-bypass exigiu `FORCE ROW LEVEL SECURITY`**
Por default, RLS NÃO se aplica ao OWNER da tabela (mesmo com policy criada). Adicionei `ALTER TABLE x FORCE ROW LEVEL SECURITY` em todas as 8 tabelas tenant-scoped. Sem isso, owner faz tudo passar.

**3. SUPERUSER bypassa RLS independente de FORCE — root cause encontrada**
Após adicionar FORCE, testes ainda retornavam todas as linhas. Investigação revelou que `POSTGRES_USER=app` cria o usuário como **SUPERUSER por default no docker postgres**. Postgres SUPERUSER bypassa RLS sempre, sem cerimônia.

**Solução arquitetural — 3 roles com responsabilidades separadas:**
- `app` (SUPERUSER) — só Prisma CLI / migrations / seed
- `app_runtime` (NOSUPERUSER, NOBYPASSRLS, DML grants) — runtime do app + testes
- `webhook_worker` (BYPASSRLS, grants mínimos) — handler webhook Meta

Testes RLS e app em runtime usam `DATABASE_RUNTIME_URL` apontando pra `app_runtime`. Migrations Prisma continuam com `DATABASE_URL` (`app`).

**4. Conflito sutil `env_file:` vs `environment:` no Compose v2**
Inicialmente o app dentro do container herdava `DATABASE_RUNTIME_URL=...localhost:5432...` do `.env.local` via `env_file:`, mesmo com `environment:` declarando override pra `postgres:5432`. Solução: removi `env_file:` dos services app/worker e listei tudo explicit em `environment:` com YAML anchor (`x-app-env: &app-env`) pra DRY.

**5. Suite de testes integration é um marco**
Primeira suite Vitest rodando contra Postgres real. Estratégia adotada:
- `tests/setup.ts` carrega `.env.local` programaticamente (sem dotenv dep)
- `tests/setup.dom.ts` separado pra `@testing-library/jest-dom` (jsdom-only)
- `tests/integration/rls.test.ts` usa `/* @vitest-environment node */` + `DATABASE_RUNTIME_URL`
- `describe.skipIf(!DB_REACHABLE)` — pula em CI sem postgres
- Fixtures isoladas por timestamp (`RLS-Test-${Date.now()}`)
- Cleanup em `afterAll` via `is_super_admin=true` + cascade delete

**6. Padrões de uso do `db.$transaction` com SET LOCAL**
`SET LOCAL app.current_condominio` exige transação ativa. Prisma `$transaction` callback garante mesma conexão pras queries dentro. Não tentar `db.$queryRaw` solto após `$executeRaw`.

**7. `current_setting('app.current_condominio', true)` — segundo argumento**
O `true` indica "não falhar se variável não existir, retorna NULL". Importante pra contextos sem RLS setado (ex: super_admin global).

**8. Senhas geradas via `openssl rand -hex 32` (64 chars hex, 256 bits de entropia)**
2 senhas separadas: `APP_RUNTIME_DB_PASSWORD` e `WEBHOOK_WORKER_DB_PASSWORD`. Ambas em `.env.local` (gitignored).

**9. Scripts npm adicionados:** `db:apply-rls` (idempotente).

## Testing Strategy

Testes desta story são **fundamentais**. RLS é difícil de testar manualmente e fácil de quebrar acidentalmente em refactors futuros. Sugestão de cenários para `tests/integration/rls.test.ts`:

| # | Cenário | Esperado |
|---|---------|----------|
| 1 | Sem context setado, query qualquer tabela tenant-scoped | retorna 0 ou erro (depende da policy) |
| 2 | Context = Cond A, SELECT unidade | retorna só unidades de A |
| 3 | Context = Cond A, INSERT setor com condominio_id de B | erro RLS |
