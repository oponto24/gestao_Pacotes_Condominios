# Story 2.4 — CRUD Morador (principal + adicionais) por unidade (admin)

**Epic:** E2 — Cadastros (terceiro CRUD tenant-scoped — fecha trilha 2.2/2.3/2.4)
**Status:** Draft
**Owner:** @dev (Dex)
**Estimativa:** 0.75 dia (mais complexo: soft delete + invariante "1 principal" + nome_normalizado + select dropdown)
**Branch sugerida:** `feature/2.4-crud-morador`
**Depende de:** 2.3 (PR #13 mergeado — pattern + Unidade existe pra select dropdown)

## User Story

**Como** admin do condomínio,
**eu quero** cadastrar/editar/desativar moradores vinculados a unidades, com 1
contato principal por unidade (fallback de notificação) + adicionais,
**para que** os porteiros associem pacotes a moradores corretos e o sistema
notifique pelo WhatsApp da pessoa certa (story 4.x).

---

## Contexto

**Terceiro CRUD tenant-scoped** — espelha 2.2/2.3 + traz **3 dores novas**:

1. **`nome_normalizado` derivado:** lowercase + remoção de acentos. Calculado
   server-side antes do save. Crítico para matching IA da story 3.7
   (algoritmo casa morador↔pacote por nome).
2. **`telefone` E.164 obrigatório + UNIQUE por condomínio:** mesma regex e
   normalização da 2.1 (`condominio.contato_telefone`). Extrair helper
   compartilhado.
3. **Invariante "apenas 1 principal por unidade":** ao setar `is_principal=true`
   em morador X, o anterior principal da mesma unidade deve virar `false`.
   Operação em transação para evitar race condition.

**Modelo `Morador` (story 1.3):**
- `condominio_id` (FK, RLS scope) + `unidade_id` (FK, on cascade delete)
- `nome` (VARCHAR 200) + `nome_normalizado` (VARCHAR 200, derivado)
- `telefone` (VARCHAR 20, E.164 obrigatório, UNIQUE por condomínio)
- `email` (VARCHAR 200, opcional, sem unique)
- `is_principal` (boolean, default false)
- `ativo` (boolean, default true) + `deleted_at` (DateTime?, soft delete LGPD)

**Diferenças vs Setor/Unidade:**
- **Soft delete** via `deleted_at` (NFR-031 LGPD) — preserva histórico de pacotes/notificações. Não DELETE físico.
- **Telefone normalizado E.164** — extrair `normalizePhone` helper de `validators/condominio.ts` para reuso
- **Invariante 1-principal-por-unidade** — lógica de negócio em transação
- **Form com Select dropdown de Unidade** — componente shadcn novo (`select.tsx`)

**Esta story:**
1. Helper `normalizarNome()` (lowercase + sem acento)
2. Refactor: extrair `normalizePhone` + regex E.164 para `src/lib/validators/_shared.ts`
3. Validators Zod com telefone E.164 + Unidade ID UUID
4. DB helpers `withTenant()` + transação para invariante principal
5. 5 endpoints REST com soft delete (DELETE seta `deleted_at`)
6. Componente shadcn `Select` (Radix Select) — novo
7. UI substitui stub `/admin/moradores`
8. Reusa: AdminLayout, EmptyState, handleApiError, errors

**Out of scope:**
- LGPD: export/anonimização de dados pessoais (story dedicada quando regulatório virar urgência)
- Bulk import (CSV — story 2.5/2.6)
- Histórico de mudanças de telefone (audit_log — story 8.3)
- Validação real de telefone via WhatsApp opt-in (story 4.x)
- Foto/avatar do morador
- Múltiplos telefones por morador (modelo é 1:1)
- Campos LGPD adicionais (CPF, RG, data nascimento) — futuro

## Notas para o @dev

### Decisões pré-aprovadas

- **`normalizarNome` em `src/lib/text/normalize.ts`** (não em validator) — é util geral, vai ser usado também em matching IA da 3.7
- **`_shared.ts` em validators** — refactor não-breaking, apenas movendo código existente da 2.1
- **Invariante 1-principal em transação:** `db.$transaction(async (tx) => { tx.morador.updateMany({ where: { unidade_id, is_principal: true }, data: { is_principal: false }}); tx.morador.update/create(...) })`. Race condition virtualmente impossível em prática (admin só faz 1 ação por vez), mas transação é correto por princípio.
- **PATCH NÃO aceita `unidade_id`:** Zod schema simplesmente não tem o campo. Mudança de unidade é exceção rara — admin cria novo morador na nova unidade + arquiva o antigo.
- **Soft delete LGPD:** `deleted_at IS NOT NULL` esconde da lista por padrão. Restore reverte para `ativo=true` mas `is_principal=false` (admin precisa re-marcar explicitamente).
- **Lista ordenada por `unidade_id, is_principal DESC, nome ASC`:** agrupa por unidade, principal primeiro, depois alfabético.

### Pontos de atenção

1. **`nome_normalizado` é coluna obrigatória no schema** — DEVE ser preenchida sempre (server-side, antes do INSERT/UPDATE). Test integration valida via re-leitura.

2. **Invariante 1-principal em CREATE:** primeira inserção pode ser `is_principal=true` sem desativar nada (não há principal ainda). Lógica do helper: SEMPRE rodar `updateMany({ unidade_id, is_principal: true }, { is_principal: false })` ANTES — se 0 rows afetados, OK.

3. **Invariante 1-principal em PATCH:** só aplicar se `data.is_principal === true` E o morador atual tinha `is_principal=false`. Se já era principal e está sendo desmarcado (`false`), nenhuma transação extra necessária.

4. **DELETE soft preserva FKs:** pacotes históricos têm `destinatario_id` apontando para morador. Soft delete não quebra (FK ainda válida).

5. **Telefone UNIQUE por condomínio:** mesmo telefone em condomínios diferentes coexiste (raro mas possível — pessoa morando em 2 condomínios com mesmo número).

6. **Test integration de invariante** é o mais crítico — sem isso, regressão futura quebraria tudo silenciosamente.

7. **Select component novo:** validar acessibilidade (keyboard nav, ESC fecha) — Radix garante, mas confirmar via smoke.
