## User Story

**Como** operador do sistema em primeira instalação,
**eu quero** rodar `npm run prisma:seed` (ou `make seed`) e obter um banco
operacional com **super-admin reconciliado ao Clerk + 1 whatsapp_number
placeholder + RLS preservado**, idempotente em qualquer ambiente (dev/staging/prod),
**para que** consiga acessar a aplicação como super_admin sem hack manual no DB
e tenha a estrutura mínima pronta antes do Epic 2 (cadastros).

---

## Contexto

`prisma/seed.ts` já existe (criado opportunisticamente nas stories 1.3/1.5)
mas tem 3 limitações:

1. **`clerk_id: 'placeholder_clerk_id_super_admin'` fica preso.** Quando o
   super-admin (Gustavo) loga via Clerk, o webhook (story 1.5) cria um
   **segundo registro** com clerk_id real porque a chave `clerk_id` é
   única. Resultado: 2 linhas para o mesmo email, super-admin "fantasma"
   sem clerk_id válido.

2. **Usa `console.log`** — viola padrão da story 1.9 (todos logs via Pino).

3. **Sem documentação operacional.** Não existe runbook explicando *quando*
   rodar (só primeira instalação? sempre? em CI?), *como* customizar
   (env `SUPER_ADMIN_EMAIL`), nem *como recuperar* se o seed falhar
   parcialmente.

Esta story:
1. Refatora `seed.ts` para usar logger Pino estruturado
2. Adiciona **lógica de reconciliação** super-admin: se já existe user
   com mesmo `email`, atualiza `role: super_admin` em vez de criar duplicata
3. Implementa **upsert resiliente** do `WhatsAppNumber` placeholder
4. Adiciona script `npm run db:reset-and-seed` para devs (drop + migrate + seed)
5. Cria `docs/runbooks/seed.md` (quando rodar, como customizar, recuperação)
6. Suite integration valida idempotência + reconciliação

**Out of scope:**
- Seed de condomínios fixture (admin/porteiro de teste) — `infra/scripts/promote-user.sh` já cobre dev; Epic 2 cria via UI
- Seed de pacotes/eventos demo — virá em story dedicada de demo data se necessário
- Seed de templates Meta WhatsApp — story 4.1
- Migração de dados de produção — fora do escopo desta story (one-off por DBA)

1. **Webhook Clerk e race condition:** se Gustavo loga ANTES do seed rodar, webhook cria user com `role: porteiro` default. Seed depois reconcilia para `super_admin`. **Validar:** seed deve permitir update de role mesmo quando user já existe via Clerk (não preservar role antigo nesse caso específico — super-admin sobrescreve).

## Dev Notes

**Execução:** 2026-05-06 — @dev (Dex)

### Decisões e desvios

1. **`db:reset-and-seed` script:** implementado como cadeia de scripts npm com guard explícito:
   ```json
   "db:guard-not-prod": "node -e \"if(process.env.NODE_ENV==='production'){...exit(1)}\"",
   "db:reset-and-seed": "npm run db:guard-not-prod && prisma migrate reset --force && npm run db:apply-rls"
   ```
   Inclui `db:apply-rls` no encadeamento (não estava no plano original) — necessário porque
   `migrate reset --force` recria o schema do zero, perdendo as roles `app_runtime`/`webhook_worker`.

2. **Webhook Clerk fallback:** implementado com **3 caminhos explícitos** (não 2 como originalmente planejado):
   - `byClerkId` existe → update normal (caminho mais comum, refresh)
   - `byEmail` existe (clerk_id é placeholder/diferente) → adopta registro, substitui clerk_id por real
   - Nenhum match → create novo
   
   Campo `reconciled_from: 'clerk_id' | 'email' | 'new'` em todos os logs facilita debug.

3. **Tratamento de string vazia em env vars:** `.env.local` tem `META_PHONE_NUMBER_ID=` (vazio).
   `??` operator preserva `''` como valor válido — bug que apareceu nos testes (criou WhatsApp
   com phone_number_id `''`). Fix: `process.env.META_PHONE_NUMBER_ID?.trim() || undefined`.
   Lição: sempre usar `|| undefined` para env vars que podem vir como string vazia.

4. **Tests via `npx prisma db seed` (não `npx tsx`):** prisma CLI não lê `.env.local` por padrão.
   Solução: passar `DATABASE_URL` explicitamente no `env` do `execSync`. Documentado no comentário
   do helper `runSeed()`.

5. **Sem `make seed`:** referência no User Story removida (não existe Makefile). Comandos npm
   são autoritativos: `npm run prisma:seed` e `npm run db:reset-and-seed`.

### 🔴 Issue HIGH — clerk_id placeholder literal causa unique constraint violation

`prisma/seed.ts:50` usa `clerk_id: PENDING_CLERK_LINK` (string literal `'pending_clerk_link'`).
Como `clerk_id` é UNIQUE no schema, rodar o seed para um SEGUNDO email diferente
(ex: trocar `SUPER_ADMIN_EMAIL=outro@example.com`) quando já existe um super-admin
pending falha:

```
$ SUPER_ADMIN_EMAIL=qa-smoke@example.com npx prisma db seed
[ERROR] seed falhou
  err: "Unique constraint failed on the fields: (clerk_id)"
```

Bug não pego pelos 4 testes porque todos usam o mesmo `SEED_EMAIL`.

**Fix sugerido (~5 linhas):**
```ts
import crypto from 'node:crypto';
const emailHash = crypto.createHash('sha256').update(email).digest('hex').slice(0, 16);
clerk_id: `pending_clerk_link_${emailHash}`,
```

E adicionar test cobrindo "rodar seed com 2 emails diferentes consecutivos não quebra".
