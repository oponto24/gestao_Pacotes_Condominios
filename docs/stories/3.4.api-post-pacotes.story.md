## User Story

**Como** PWA da portaria,
**eu quero** enviar a foto da etiqueta + código (opcional) capturados,
**para que** o sistema crie um rascunho de pacote, salve a foto e enfileire
o job de IA que vai extrair os dados.

---

## Contexto

**FR-012 (PRD):** "Sistema deve obrigatoriamente capturar uma foto da etiqueta."
**FR-013:** "Sistema deve enviar a foto para pipeline de IA (Claude Haiku 4.5)."
**FR-019:** "Registrar horário de chegada automático (timestamp do servidor)."
**FR-020:** "Registrar funcionário responsável (capturado via login)."

**Esta story entrega APENAS o backend.** A foto vem do `<PhotoCapture>` (3.2)
e o código do `<BarcodeScannerInput>` (3.3) via FormData. Worker IA é story 3.5.

**Fluxo:**
1. UI envia `POST /api/pacotes` multipart com `file` (Blob JPEG) + `codigo_rastreio?`
2. API: `requirePorteiro()` → valida tenant + role
3. Cria `pacote_id` UUID
4. Salva foto em `storage` no path `{condominio_id}/{pacote_id}/original.jpg`
5. Em **transação** (`withTenant`):
   - `pacote.create({ status: 'rascunho', codigo_rastreio?, ... })`
   - `pacote_foto.create({ pacote_id, storage_path, mime_type, bytes, hash_sha256, is_principal: true })`
   - `pacote_evento.create({ pacote_id, tipo: 'criado', user_id })`
6. Enfileira job `extractLabel` com `{ pacote_id, condominio_id }`
7. Retorna `{ pacote_id, foto_storage_path }` → UI redireciona para `/chegada/confirmar/[pacote_id]` (story 3.8)

## Notas para o @dev

### Decisões pré-aprovadas

1. **Path `/api/pacotes` (não `/api/admin/pacotes`):** porteiro acessa, não admin. Convenção: `/api/admin/*` é só pra admins. Endpoints da portaria ficam em `/api/pacotes`, `/api/retirada`, etc.
2. **`requirePorteiro` da 3.1** já aceita admin como substituto — não precisa criar guard novo.
3. **Storage path determinístico:** `{cond}/{pacote_id}/original.{ext}`. Permite cleanup fácil (admin deleta pacote → deleta dir inteiro).
4. **Hash SHA-256** da foto guardado em `pacote_foto.hash_sha256` — futura deduplicação ou audit forense (NFR-033 retenção 12 meses).
5. **Enqueue FORA da transação:** se Redis down, pacote ainda é criado. Re-enfileiramento manual via tela admin futura.
6. **Multipart com Blob:** UI envia `FormData` com `file` field. `request.formData()` no route handler.
7. **Limite 5MB:** PhotoCapture gera ~300KB típico. 5MB cobre fotos extra grandes do iPhone Pro sem comprimir. Defesa contra abuso.
8. **`pacote_evento.tipo = 'criado'`:** verificar enum exato no schema (`PacoteEventoTipo`). Se nome diferente, usar o que existe.

### Pontos de atenção

1. **`request.formData()` é assíncrono** + parse de multipart é pesado em uploads grandes. Por isso limite 5MB.
2. **Storage é síncrono in-memory para `LocalStorageDriver`:** `storage.put()` retorna após arquivo no disco. Para S3/R2 futuro vai ser async.
3. **Hash via stream** se arquivo for grande — mas com 5MB max, lê tudo em buffer é OK.
4. **`pacote_foto.condominio_id`:** redundante com `pacote.condominio_id` mas necessário pra RLS direto na tabela. Setar igual.
5. **Transação Prisma rolla back arquivo no storage?** NÃO — storage não é transacional. Se DB falha após `storage.put`, arquivo fica órfão. Mitigação: cleanup job futuro (story 8.x) detecta arquivos sem `pacote_foto` correspondente. Para MVP, aceitar.
6. **`enqueue` retorna o job ID:** logar para correlação com `pacote_id`.

### Não fazer agora

- Worker IA `extractLabel` real → 3.5
- Tela `/chegada/confirmar/[id]` → 3.8
- Algoritmo matching CEP+nome → 3.7
- Anti-duplicação por código de rastreio → 3.4 não checa, IA process pode flagar
- Cleanup de fotos órfãs → tech debt 8.x
- Compressão server-side da foto → cliente já comprime
- Múltiplas fotos por pacote → MVP só `is_principal: true`

## Dev Notes

### Pattern reusado
- `withTenant` da 1.6 (Epic 2 estabeleceu)
- `requirePorteiro` da 3.1
- `storage.put` da 1.9
- `enqueue` da 1.8
- `handleApiError` padrão API

### Nova superfície
- **Primeiro endpoint da portaria** (não admin) — `/api/pacotes`
- **Primeiro upload multipart** com Blob no projeto
- **Primeira transação que toca 3 tabelas + storage + queue** — pattern de referência pra próximas stories que combinam DB + I/O externo

### Riscos

1. **Storage I/O síncrono:** se disco encher, `storage.put` falha. Logar bem.
2. **Race condition foto vs DB:** ordem importa: foto ANTES de DB. Se DB falha, arquivo fica órfão (aceitável MVP).
3. **Job duplicado:** se cliente reenviar (ex: timeout), 2 pacotes criados. Mitigação: idempotency-key futura (header `Idempotency-Key`). Para MVP, aceitar duplicata visível na lista admin.
4. **Tamanho do payload no Next.js:** Next 14+ Body limit padrão é 1MB. Para multipart 5MB, configurar `bodySizeLimit` em `next.config.ts` ou via route handler.
