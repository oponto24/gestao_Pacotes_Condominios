## User Story

**Como** desenvolvedor do projeto,
**eu quero** um middleware aplicacional que extrai `condominio_id` do user autenticado (via Clerk) e injeta automaticamente como contexto RLS Postgres em cada query,
**para que** queries Prisma feitas em rotas protegidas sejam **automaticamente filtradas pelo condomínio do usuário** sem precisar passar `condominio_id` em cada `where`, e a defesa em profundidade (middleware + RLS) funcione na prática em runtime — não só nos testes integration.

---

## Contexto

A 1.4 entregou RLS funcionando matematicamente (validado por suite com 6 testes), mas exige que **cada conexão DB tenha `app.current_condominio` setado** antes de qualquer query. A 1.5 entregou `getCurrentUser()` que devolve `{ user: { condominio_id, role }, ... }`.

Esta story conecta os dois: cria abstração de "tenant context" que:
1. Resolve `condominio_id` do user atual (via getCurrentUser)
2. Wrappa queries Prisma em transação
3. Executa `SET LOCAL app.current_condominio = '<uuid>'` antes da query real
4. Para `super_admin` (sem condomínio) seta `app.is_super_admin = true`
5. Falha clara se não autenticado mas tentando acessar dados tenant-scoped

**Heads-up crítico do QA da 1.5:** middleware roda em CADA request. Performance importa. Cache `getCurrentUser` por lifetime de request via `React.cache()` (built-in do React 19) é a estratégia adequada — Redis seria over-engineering pra MVP.

**Out of scope:**
- UI pra associar user a condomínio (story 2.x — PO/admin promove user)
- Roles UI (story 2.x)
- Cache cross-request via Redis (otimização futura)
- Webhook handler já tem `db` direto (não passa pelo middleware tenant) — correto, é cross-tenant por design

## Notas para o @dev

### Sobre a decisão Prisma $extends vs withTenant helper

**Opção A — Prisma Client Extension (`$extends`):**
```typescript
const tenantDb = baseDb.$extends({
  query: {
    $allOperations({ args, query }) {
      return baseDb.$transaction(async (tx) => {
        await tx.$executeRawUnsafe(`SET LOCAL app.current_condominio = '${condominioId}'`);
        // re-execute via tx... complicated
      });
    },
  },
});
```
**Vantagem:** call-sites usam Prisma normalmente (`db.unidade.findMany()`)
**Desvantagem:** complexo de implementar — não dá pra reusar a mesma transação entre o `SET LOCAL` e a query original direto via extension. Pode requerer hack.

**Opção B — `withTenant(condominioId, callback)`:**
```typescript
await withTenant(condominioId, async (tx) => {
  return tx.unidade.findMany();
});
```
**Vantagem:** simples, transparente, usa Prisma `$transaction` natural
**Desvantagem:** call-sites precisam wrappar — mais verboso

**Minha sugestão (não obrigatória):** **B** pra MVP. É menos elegante mas previsível. Quando virar dor real, refatora pra extension.

### React.cache() para deduplicar getCurrentUser

```typescript
import { cache } from 'react';
export const getTenantContext = cache(async () => {
  // ... resolve user, condominio_id, etc
});
```

`cache()` do React (não confundir com `unstable_cache` do Next) é por-request automaticamente. Múltiplas chamadas no mesmo render server retornam o mesmo resultado.

### Webhook handler NÃO usa middleware tenant

Importante: `/api/webhooks/clerk` é cross-tenant por design (precisa criar/atualizar users de qualquer condomínio). Continua usando `db` (não `dbInTenantContext`). Mesmo padrão pro webhook Meta (story 4.4).

### Decisões pré-aprovadas

- 3 erros como classes (não Result type) — convenção Next.js Server Actions é throw + catch
- Endpoints `/api/me` e `/api/me/data` SÃO smoke tests permanentes — ficam na app
- Script `promote-user.sh` aceita email como arg, sem fancy CLI
- `is_super_admin = true` ao detectar `role === 'super_admin'` no middleware (não usar `current_condominio` nesse caso)

### Pontos de atenção

1. **`SET LOCAL` exige transação ativa.** Implementação correta é `$transaction(async (tx) => { await tx.$executeRaw\`SET LOCAL ...\`; return tx.... })`. Fora de transação, `SET LOCAL` é silenciosamente ignorado.

2. **SQL injection em `SET LOCAL`:** o `condominio_id` vem de DB (não user input direto), mas ainda assim usar parameterized query: `await tx.$executeRaw\`SET LOCAL app.current_condominio = ${condominioId}::uuid\``. Prisma trata escape automaticamente.

3. **Connection pooling:** `SET LOCAL` vale só pra transação. Próxima request começa com contexto limpo. Não há vazamento entre requests.

4. **Performance:** `dbInTenantContext` chama `getTenantContext()` que é cached. Mas a transação em si tem overhead pequeno (~5ms). Se virar gargalo, otimização futura é Redis cache do user.

5. **Erro `PendingProvisioningError`:** acontece nos primeiros ms após signup (Clerk autentica antes do webhook chegar). Frontend já trata via `getCurrentUser()` retornando `kind: 'pending_provisioning'`. Esta story só replica o tratamento no middleware.

6. **`promote-user.sh` é DEV ONLY.** Em produção, admin promove via UI (story futura). Documentar.

### Não fazer agora

- ❌ UI de associação user→condomínio (story 2.x)
- ❌ UI de troca de role (story 2.x)
- ❌ Cache Redis cross-request do user (otimização futura)

## Dev Notes

### Decisões e descobertas

**1. T5 — Decisão arquitetural: `withTenant(callback)` (Opção B), NÃO `$extends` (Opção A)**
Razão: `$extends` precisaria envolver cada query Prisma numa transação separada com `SET LOCAL` antes — mas o extension API do Prisma roda DENTRO da query, não fora. Implementações que tentam isso usam hacks (recriar Prisma client, reflexão). Opção B é explícita, previsível e funciona com a API natural de `$transaction`. Trade-off aceito: call-sites são levemente mais verbosos (`withTenant(async (tx) => tx.unidade.findMany())`).

**2. Postgres não aceita placeholder em `SET LOCAL` — workaround com regex UUID**
Tentei `tx.$executeRaw\`SET LOCAL app.current_condominio = ${ctx.condominioId}::uuid\`` (template literal Prisma). Falhou com `syntax error at or near "$1"`. Postgres simplesmente não suporta parameterized queries em `SET LOCAL`/`SET`. Solução: `$executeRawUnsafe` com interpolação de string + **validação UUID via regex ANTES** de interpolar — defesa contra injection mesmo que `condominioId` venha de fonte não confiável (na prática vem do banco, mas defense-in-depth).

**3. `React.cache()` para deduplicar getTenantContext**
Múltiplas chamadas a `getTenantContext()` no mesmo request server retornam o mesmo resultado sem re-fetch (Clerk auth + DB lookup do user). Importante porque qualquer Server Component / route handler / loader pode chamar — sem cache, seria N+1 de auth lookups por request.

**4. `super_admin` precisa LIMPAR `app.current_condominio` ao setar `is_super_admin`**
Sem isso, valor antigo de pool de conexão poderia "vazar" e a policy avaliaria com tenant_id stale. Implementei: ao setar `is_super_admin=true`, primeiro limpa `current_condominio = ''`. Idem inverso (tenant context limpa `is_super_admin=''`).

**5. Webhook Clerk continua usando `db` direto (NÃO `withTenant`)**
Por design: webhooks são cross-tenant. `getTenantContext()` lançaria `UnauthorizedError` (sem auth Clerk no request) e quebraria o handler. Documentei explicitamente em SCHEMA.md §12.

**6. Erros tenant com `httpStatus` — mapeamento direto pra resposta**
3 classes com código + status: 401 (unauthorized), 403 (no_condominio_assigned), 503 (pending_provisioning). Helper `isTenantError(e)` permite catch type-safe nos route handlers.

**7. Smoke endpoints `/api/me` e `/api/me/data` ficam permanentes**
Não são teste descartável — são dogfooding contínuo. Toda story futura pode usar pra validar que tenant context segue funcionando após mudanças.

**8. `promote-user.sh` é DEV ONLY**
Cria condomínio fixture "Edifício Teste 1.6" + promove user a admin. Idempotente. Não usar em prod — em prod, admin associa via UI (story futura). Documentei em `infra/README.md`.

**9. Ordem dos commits importa pra cleanup de fixtures**
`afterAll` dos testes integration usa `app` (SUPERUSER) pra deletar fixtures, com `SET LOCAL app.is_super_admin = 'true'` por defensive coding (mesmo que `app` não precise — bypassa RLS).
