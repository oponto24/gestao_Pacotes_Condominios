# Story 10.4 — Bifurcação fluxo chegada (porteiro vs admin)

**Epic:** E10 — Hierarquia operacional
**Status:** Done (cerimônia inline pelo Orion)
**Owner:** @dev (Dex)
**Branch:** `feature/10.4-bifurcacao-fluxo-chegada`
**Depende de:** 10.1, 10.2, 10.3
**Bloqueia:** 10.5 (UI /administracao/organizar)

## Mudanças

- `pacote-confirmar.ts`: lê `condominio.tem_administracao`. Se true → status pós-confirmação vira `aguardando_organizacao` em vez de `rascunho`. Retorna flag pro cliente.
- `IAExtractionForm.tsx`: redirect baseado em `tem_administracao` da resposta. Sem admin: `/chegada/organizar/{id}`. Com admin: `/chegada?msg=enviado_administracao` (próximo pacote).
- `/chegada/organizar/[id]/page.tsx`: bloqueio defensivo se status for `aguardando_organizacao` (caso porteiro acesse URL diretamente).
- Audit log do evento `confirmado` ganha `tem_administracao` no metadata.

## Comportamento

**Sem administração** (default):
1. Porteiro confirma morador → status `rascunho`
2. Porteiro escolhe setor/posição → status `aguardando_retirada` + WhatsApp dispara

**Com administração**:
1. Porteiro confirma morador → status `aguardando_organizacao` (pacote some da fila do porteiro)
2. **[Story 10.5]** Admin acessa `/administracao/organizar` → escolhe setor/posição → status `aguardando_retirada` + WhatsApp dispara

## Validações

- typecheck/lint/build clean
- 300/300 unit tests verde

## Out of scope (vai pra 10.5)

- ❌ UI `/administracao/organizar` — pacotes em `aguardando_organizacao` ficam "stuck" sem essa tela. Em prod com `tem_administracao=true` ainda sem 10.5, fluxo quebra. **Não ativar `tem_administracao=true` em condomínio antes de 10.5 estar deployada.**
