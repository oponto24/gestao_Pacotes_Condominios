## User Story

**Como** desenvolvedor do projeto,
**eu quero** Clerk integrado ao Next.js (sign-in, sign-up, `<UserButton>`, middleware de auth via `proxy.ts`) + webhook handler que sincroniza usuários Clerk com a tabela `user` do banco,
**para que** os fluxos de cadastro/login funcionem ponta-a-ponta e cada user no Clerk tenha um registro correspondente no Postgres com `condominio_id`/`role` (necessário pra middleware de tenant context na story 1.6).

---

## Contexto

A doc OFICIAL ATUALIZADA do Clerk Next.js (passada pelo fundador) traz mudanças importantes vs material antigo:

1. **`proxy.ts`** em vez de `middleware.ts` — nome novo do arquivo
2. **`<Show when="signed-in">` / `<Show when="signed-out">`** — substituem `<SignedIn>`/`<SignedOut>` (deprecated)
3. **`clerkMiddleware()`** — substitui `authMiddleware()` (deprecated)
4. Matcher do middleware mais específico

Esta story segue a doc oficial mais recente.

**Out of scope (vem em stories futuras):**
- Middleware de tenant context (`getTenantContext`, `SET LOCAL`) → story 1.6
- Roles UI (atribuir admin/porteiro a user) → entra com cadastros (story 2.x)
- Magic link / social login (Google, etc.) → roadmap pós-MVP
- Organizations Clerk (multi-tenancy nativo Clerk) → não usamos, condomínio é nosso tenant

### Por que é necessário webhook handler

Clerk é a fonte da verdade de auth (clerk_id, email, nome). Mas precisamos espelhar para Postgres porque:
1. Middleware de tenant (1.6) precisa fazer JOIN entre `clerk_id` e `user.condominio_id`
2. RLS policies referenciam `user.id` (UUID interno), não `clerk_id`
3. Pacotes têm FK pra `user_id` (porteiro/admin que recebeu/entregou)

Webhook Clerk dispara `user.created` / `user.updated` / `user.deleted` → handler em `/api/webhooks/clerk` faz upsert na tabela `user`.

### Pontos de atenção

1. **Validação de webhook HMAC é obrigatória.** Sem isso, qualquer um pode forjar um POST e criar users falsos. `svix.Webhook` cuida disso — passa `CLERK_WEBHOOK_SECRET` no constructor.

2. **`auth()` do Clerk é async** em Next.js 15+ — `const { userId } = await auth();`.

3. **Webhook chega ANTES da page render.** Quando user faz signup, Clerk redireciona pra home, mas o webhook pode ainda não ter chegado. Por isso `getCurrentUser()` precisa do estado `pending_provisioning`. UI deve mostrar "Configurando sua conta..." nesse caso.

4. **Configurar webhook no Clerk Dashboard:**
   - URL: `http://SEU_DOMINIO/api/webhooks/clerk`
   - Em dev, expor via tunnel (ngrok / cloudflared) e usar URL do tunnel
   - Eventos: `user.created`, `user.updated`, `user.deleted`
   - Copiar `Signing Secret` → `CLERK_WEBHOOK_SECRET` no `.env.local`
   - **Documentar no Dev Notes o tunnel/URL usado**

5. **Tunnel pra dev**: `cloudflared tunnel --url http://localhost:3000` é gratuito e instantâneo. Alternativa: ngrok.

6. **Default `role` pra novos users:** definir como `porteiro` (papel mais restrito). Admin precisa promover manualmente. Documentar.

7. **`condominio_id` é NULL pra novos users** — eles ainda não estão associados a nenhum condomínio. Admin associa via UI futuramente (story 2.x). Apenas super-admin tem `condominio_id = NULL` permanente.

8. **Hot reload do Next.js + ClerkProvider:** se ver erro "ClerkProvider not found", reiniciar dev server.

## Dev Notes

### Decisões e descobertas

**1. Crítico: `proxy.ts` da doc oficial NÃO funciona com Clerk 7.3.1 — voltei para `middleware.ts`**
A doc passada pelo fundador (de Next.js 16+) recomenda `proxy.ts`, mas o Clerk 7.3.1 (último estável quando rodei `npm install @clerk/nextjs`) ainda procura por `src/middleware.ts` e lança erro explícito se não encontrar:
```
Clerk: auth() was called but Clerk can't detect usage of clerkMiddleware().
- Your middleware file exists at ./src/middleware.(ts|js)
```
Renomeei `src/proxy.ts` → `src/middleware.ts`. Quando upgradar Clerk pra versão que suporte `proxy.ts`, basta renomear de volta. **Atualizar story 1.5 e ARCHITECTURE.md em refactor futuro.**

**2. Volume `node_modules` cacheado — gotcha repetido**
Mesmo problema da story 1.3: instalei `@clerk/nextjs` no host, mas o container usava o volume `node_modules` antigo (sem Clerk). Solução conhecida: `docker volume rm gestao-pacotes-dev_node_modules gestao-pacotes-dev_next_cache && up --build`. **Adicionar isso ao playbook do `infra/README.md`.**

**3. Race condition app+worker ao recriar volume — gotcha repetido**
Subindo app e worker simultaneamente após `volume rm` dá `mkdir /.bin: file exists`. Solução: `up app` → wait → `up worker`. Já documentado na 1.3.

**4. Webhook handler retorna 200 quando `CLERK_WEBHOOK_SECRET` é placeholder**
Implementei: se `CLERK_WEBHOOK_SECRET` for vazio ou ainda for `whsec_PENDING_DASHBOARD_CONFIG`, handler retorna `200 { ok: false, reason: 'webhook_not_configured' }`. Razão: evita Clerk ficar retentando webhook pra um endpoint não configurado ainda. Quando T15 for executado e secret real chegar, comportamento muda automaticamente.

**5. Helper `getCurrentUser()` retorna 3 estados**
- `null` — não autenticado
- `{ kind: 'pending_provisioning', clerk_id }` — autenticado mas webhook ainda não criou linha (acontece nos primeiros ms após signup)
- `{ kind: 'authenticated', user }` — autenticado com user persistido

A página home trata os 3 com mensagens distintas. Importante pra UX (signup → "Configurando sua conta..." em vez de erro).

**6. Webhook valida HMAC svix corretamente**
Testado via `curl -X POST` sem headers svix → 400. Sem headers válidos → 400. Apenas com assinatura válida → processa.

**7. `user.deleted` faz `updateMany` com `ativo=false`**
NÃO `delete`. Usei `updateMany` em vez de `update` pra evitar erro se o user nunca foi criado (idempotência).

**8. Lint warning fixado durante execução**
Inicialmente o `catch (err)` deixava `err` não usado. Refatorei pra `catch` sem var.

**9. ESLint warning não-bloqueante: `next lint` deprecated**
Mesma observação de stories anteriores. Migration futura.

   - `user.created`

   - `user.deleted`

## CodeRabbit Integration

**Severity esperada:** média-alta (auth + webhook = superfície de ataque crítica).

**Pontos de atenção pra revisão:**
- Webhook **valida HMAC svix** antes de processar payload (CRITICAL se faltar)
- `CLERK_WEBHOOK_SECRET` lido de env, NUNCA hardcoded
- Webhook idempotente (upsert, não insert) — Clerk pode retentar
- Erros internos retornam 200 ao Clerk (com log), evita loop de retry
- `getCurrentUser()` não vaza `clerk_id` em respostas públicas
- Páginas sign-in/sign-up são CATCH-ALL routes (`[[...sign-in]]`)
- `proxy.ts` matcher exclui assets estáticos (não interceptar PNG, CSS, etc)

**Severidades esperadas:**
- CRITICAL: se webhook não validar HMAC ou se secret aparecer em arquivo committed
- HIGH: se `user.deleted` deletar linha em vez de marcar `ativo=false`
- MEDIUM: se `getCurrentUser` faltar tipagem ou logs vazarem dados sensíveis
