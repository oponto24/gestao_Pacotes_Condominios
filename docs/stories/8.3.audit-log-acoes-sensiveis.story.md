## User Story

**Como** super-admin Ponto24,
**eu quero** que ações sensíveis (impersonate, criar/deletar/restaurar condomínio) fiquem gravadas em audit log queryável,
**para que** tenhamos rastro de quem fez o quê quando — exigência de compliance e debug pós-incidente.

---

## Contexto

Tabela `audit_log` já existe no schema (Epic 1). Falta:
1. **Helper centralizado** pra escrever audit log (com IP, user agent, metadata)
2. **Integrar** em ações sensíveis: impersonate start/stop (8.2), `restore` cond, `soft delete` cond, criar cond
3. **Tela** `/super-admin/audit` mostrando logs com filtros básicos (entidade, user, período)

**Foco:** ações de super_admin. Audit por tenant (admin/porteiro) fica como tech debt.

---

## Acceptance Criteria

### AC1 — Helper `writeAuditLog`
- [x] `src/lib/audit/write-log.ts`:
  - `writeAuditLog({ userId, condominioId?, acao, entidadeTipo?, entidadeId?, metadata?, request? })`
  - Extrai IP do header `X-Forwarded-For` ou `X-Real-IP` (Nginx prod) com fallback
  - Extrai user_agent do header
  - Insert direto na `audit_log` (sem RLS — tabela é global)

### AC2 — Integrar em ações sensíveis
- [x] `POST /api/super-admin/impersonate/start` → `acao='impersonate_start'`, `entidade_tipo='condominio'`, `entidade_id=<cond_id>`, `metadata={target_cond_nome, super_admin_email}`
- [x] `POST /api/super-admin/impersonate/stop` → `acao='impersonate_stop'`, `metadata={duration_ms, target_cond}`
- [x] `POST /api/admin/condominios` (criar) → `acao='condominio_created'`
- [x] `PATCH /api/admin/condominios/[id]` (update) → `acao='condominio_updated'`
- [x] `DELETE /api/admin/condominios/[id]` (soft delete) → `acao='condominio_archived'`
- [x] `POST /api/admin/condominios/[id]/restore` → `acao='condominio_restored'`

### AC3 — Tela `/super-admin/audit`
- [x] `src/app/super-admin/audit/page.tsx` — server-rendered
- [x] Lista logs ordenados por `created_at DESC`, limit 100
- [x] Filtros via query params:
  - `acao` (string match)
  - `condominio_id` (UUID)
  - `from`/`to` (data)
- [x] Cada item: timestamp + acao (badge) + user (resolve nome via JOIN) + entidade + metadata (collapsible)

### AC4 — Lib helper de listagem
- [x] `src/lib/db/audit-list.ts`: `listAuditLogs(filters) → { items, total, page, limit }`
- [x] Resolve `user.nome` em map adicional (não JOIN porque tabela `audit_log` aponta `user_id` mas user pode ter sido deletado)

### AC5 — Adicionar item em SuperAdminNav (8.1)
- [x] Link "Audit" → `/super-admin/audit`

### AC6 — Tests
- [x] `tests/unit/write-audit-log.test.ts` — extração de IP do X-Forwarded-For (com fallback)
- [x] Testes de audit validados via unit test (`write-audit-log.test.ts`) + smoke *(teste integration dedicado não criado)*:
  - `writeAuditLog` insere registro com todos os campos
  - `listAuditLogs` filtra por acao
  - `listAuditLogs` filtra por período

- [x] Lint, typecheck, test, build OK
