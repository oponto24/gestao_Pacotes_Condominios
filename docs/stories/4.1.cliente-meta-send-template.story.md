## User Story

**Como** sistema,
**eu quero** um cliente HTTP encapsulado da Meta Cloud API com helper `sendTemplate(...)`,
**para que** workers e handlers (4.3) possam disparar templates aprovados sem replicar boilerplate de auth, base URL, error handling.

---

## Contexto

**FR-035 (PRD):** "Sistema deve usar templates Meta aprovados (categoria utility), com variáveis para nome do condomínio, nome do morador, setor."
**CON-003:** "WhatsApp via Meta Cloud API direto, sem BSP, no MVP."
**CON-005:** "Modelo de WhatsApp: número único compartilhado multi-tenant."

**Setup Meta concluído (Etapas 1-5 do runbook `docs/runbooks/setup-meta-whatsapp.md`):**
- App `Gestao de Pacotes Condominios` (App ID `2194393834738443`)
- WABA sandbox `1446111150585784` + Phone Number ID sandbox `1138774412646193`
- Token permanente `META_ACCESS_TOKEN` no `.env.local`
- Template `pacote_chegou` submetido pra revisão Meta em 2026-05-08 16h25 (em análise)

**Esta story entrega APENAS o cliente HTTP.** Geração do QR (4.2), worker BullMQ (4.3) e webhook (4.4) ficam em stories separadas.

**Endpoint alvo:**
```
POST https://graph.facebook.com/v25.0/{phone_number_id}/messages
Authorization: Bearer {access_token}
Content-Type: application/json
```

**Fluxo:**
1. Caller chama `sendTemplate({ to, templateName, languageCode, components })`
2. Cliente monta payload Meta válido (`messaging_product: "whatsapp"`, `type: "template"`, ...)
3. Faz POST com fetch nativo, timeout 15s
4. Em sucesso retorna `{ wamid, contacts }`
5. Em erro Meta (4xx/5xx), parseia `error.code` e lança `MetaApiError` tipada

### AC3 — Helper `sendTemplate`
Assinatura:
```ts
sendTemplate(input: {
  to: string;                    // phone E.164 sem + (ex: 5511999999999)
  templateName: string;          // ex: 'pacote_chegou'
  languageCode: string;          // ex: 'pt_BR'
  headerImageUrl?: string;       // URL pública (futuro: upload via media endpoint)
  bodyParams: string[];          // ex: ['João Silva', 'Edifício Central']
}): Promise<{ wamid: string; mock?: boolean }>
```

- [x] Monta payload Meta:
```json
{
  "messaging_product": "whatsapp",
  "to": "5511999999999",
  "type": "template",
  "template": {
    "name": "pacote_chegou",
    "language": { "code": "pt_BR" },
    "components": [
      { "type": "header", "parameters": [{ "type": "image", "image": { "link": "..." } }] },
      { "type": "body", "parameters": [
        { "type": "text", "text": "João Silva" },
        { "type": "text", "text": "Edifício Central" }
      ]}
    ]
  }
}
```
- [x] Omite `header` component se `headerImageUrl` ausente
- [x] `fetch` com timeout 15s via `AbortController`
- [x] Retorna `{ wamid: response.messages[0].id }` (Meta chama `wamid`)

### AC4 — Tratamento de erros — `MetaApiError`
Criar classe `MetaApiError extends Error` em `src/lib/meta-whatsapp/errors.ts` com:
- `code: number` (Meta error code)
- `subcode?: number`
- `httpStatus: number`
- `fbtrace_id?: string`
- `retriable: boolean` — true para `131026` (re-engagement), `131056` (rate), `5xx`, network
- `userFacing: string` — mensagem traduzida pra UI ("Mensagem fora da janela de 24h", "Número inválido", etc.)

Mapping mínimo (cobrir os 5 códigos mais comuns, resto cai em fallback genérico):
| Meta code | retriable | userFacing |
|-----------|-----------|------------|
| 131026 | false | Morador não tem WhatsApp ativo neste número |
| 131047 | false | Janela de 24h expirada — só template é permitido |
| 131056 | true | Limite de envios atingido, retry mais tarde |
| 100 | false | Parâmetro inválido — confira número e template |
| network/5xx | true | Falha temporária Meta — retry automático |
