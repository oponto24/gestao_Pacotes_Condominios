## User Story

**Como** sistema,
**eu quero** que o worker pegue o job `extractLabel` enfileirado pela 3.4,
baixe a foto do storage, mande pro Claude Haiku 4.5 vision e extraia os
dados estruturados da etiqueta,
**para que** o porteiro não precise digitar nada — só revisar os dados
extraídos (story 3.8) antes de gravar.

---

## Contexto

**FR-013 (PRD):** "Sistema deve enviar a foto para pipeline de IA (Claude
Haiku 4.5 vision) e extrair em JSON estruturado: nome do destinatário,
endereço, CEP, complemento (apto/bloco), transportadora."
**CON-004 (PRD):** "LLM obrigatório: **Anthropic Claude Haiku 4.5** com
prompt caching (não usar GPT/Gemini)."
**NFR-041 (PRD):** "Custo de IA por etiqueta: <R$ 0,05 (com prompt caching)."

**Esta story entrega APENAS o worker.** UI de confirmação fica na 3.8.
Algoritmo de matching CEP+nome → unidade fica na 3.7.

**Fluxo:**
1. Worker BullMQ recebe job `extractLabel` com `{ pacote_id, condominio_id }`
2. Carrega `pacote_foto.is_principal=true` via Prisma direto (sem RLS — worker_role)
3. Lê arquivo do storage via `storage.get(storage_path)`
4. Chama Anthropic Messages API com:
   - `model: 'claude-haiku-4-5'`
   - `system`: prompt longo com schema + instruções (com `cache_control: ephemeral`)
   - `messages`: imagem em base64 + instrução curta
5. Recebe resposta JSON, valida via Zod
6. Atualiza pacote: `ia_extracao_raw`, `ia_confianca`, `ia_processada_em`
7. Cria evento `pacote_evento` tipo `ia_processou`
8. Status do pacote permanece `rascunho` (story 3.7/3.8 mudam depois)

## Notas para o @dev

### Decisões pré-aprovadas

1. **Modelo `claude-haiku-4-5-20251001`** (mais novo Haiku 4.5 estável). Não usar `latest` — pin na versão pra evitar regressão silenciosa.
2. **Prompt caching ephemeral** (~5min TTL): cachear system prompt completo (schema + exemplos). Custo cache miss ~25% mais caro mas hit é ~10% do custo input. ROI positivo se >2 etiquetas processadas em 5min (típico em condomínio movimentado).
3. **`max_tokens: 500`** na resposta — JSON estruturado curto, 500 é folga.
4. **`temperature: 0`** — extração estruturada precisa ser determinística, não criativa.
5. **Vision via base64** (não URL pública) — segurança + não precisa expor storage.
6. **Schema Zod no parsing**, não JSON Schema do Anthropic — Anthropic não tem function calling nativo como OpenAI; manda JSON em texto e Zod valida. Se IA mandar markdown ou texto extra, regex extrai bloco JSON.
7. **`status` do pacote NÃO muda aqui** — fica `rascunho`. Story 3.7 (matching) ou 3.8 (admin confirma) movem para `pendente_identificacao` ou outro.
8. **Worker bypassa RLS** com `app.is_super_admin = 'true'` (já configurado em `withTenantContext` quando `kind: 'super_admin'`). Filtra por `condominio_id` do payload pra não vazar.

### Pontos de atenção

1. **Buffer de foto pode ser grande** (até 5MB) — base64 dobra pra ~7MB no payload. Anthropic aceita até 5MB de imagem (após base64). Se foto >5MB, **comprimir antes** ou rejeitar (PhotoCapture já comprime para 1280px / ~300KB).
2. **`media_type` do bloco vision** precisa ser exato: `image/jpeg` ou `image/png`. Lê de `pacote_foto.mime_type`.
3. **Resposta da IA vem em texto** — usa `message.content[0].text` (assumindo block do tipo text). Sempre verificar tipo.
4. **Anthropic pode retornar JSON em markdown** (` ```json `). Strip markdown antes do parse: `text.replace(/^```json\s*/, '').replace(/```$/, '')`.
5. **Rate limit de Tier 1**: 50 requests/min. Burst de 5+ pacotes em sequência pode disparar 429. BullMQ retry com backoff resolve.
6. **Custo:** Input ~1500 tokens (system + schema + exemplos) + ~500 tokens (image) + ~200 tokens output. Com cache hit: ~$0.003/foto = ~R$0.015. **Atinge NFR-041 com folga**.
7. **Logs sensíveis:** NÃO logar `ia_extracao_raw` completo (pode conter dados pessoais do destinatário). Logar só `confianca` e `tokens_used`.

### Não fazer agora

- Algoritmo de matching CEP+nome → 3.7
- UI de confirmação `/chegada/confirmar` → 3.8
- Pipeline alternativa fallback (GPT-4) → fora MVP (CON-004 obriga Anthropic)
- Treinar modelo customizado → fora MVP
- Cache de respostas de fotos similares → tech debt
- Observabilidade detalhada (Sentry, Langfuse) → Epic 8
- Quotation per-tenant (rate limit por condomínio) → Epic 8

## Dev Notes

### Pattern reusado
- Pattern de jobs do BullMQ (1.8) — `extract-label.ts` espelha `ping.ts`
- `storage.get` da 1.9
- Logger estruturado com `loggerForJob`
- Prisma direto (sem withTenant) — worker tem permissão super_admin

### Nova superfície
- **Primeira integração com LLM** (Anthropic Claude Haiku 4.5)
- **Primeira chamada externa de API** com custo real ($)
- **Prompt caching** — pattern replicável em stories futuras (4.x WhatsApp templates, 7.x parsing código ML)
- **`src/lib/anthropic/`** — virá uma família de prompts (`extract-label`, futuro `extract-codigo-ml`, etc)

### Riscos

1. **API key vazada:** rotacionar imediatamente. Documentar no runbook que `.env.local` é gitignored e produção usa secret manager (Hostinger/VPS env via systemd).
2. **Custo descontrolado:** monitorar via Anthropic Console diariamente. Adicionar alerta budget. Implementar rate limit por tenant em Epic 8.
3. **Confiança alta mas dados errados:** IA pode "alucinar" com convicção. Daí o **modo "sempre confirmar"** (FR-014) — admin valida antes de gravar. Mitigado pela story 3.8.
4. **Imagem pequena/borrada:** `confianca` baixa sinaliza, UI da 3.8 destaca em amarelo. Não-bloqueante mas operacional.
5. **Schema da IA evolui:** Anthropic pode mudar formato de resposta. Zod `safeParse` evita crash, log warning permite debug.
