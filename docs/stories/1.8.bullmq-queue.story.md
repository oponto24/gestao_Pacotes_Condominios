## Contexto

A 1.2 entregou worker como placeholder (`workers/index.mjs` com setInterval). A 1.7 instalou `ioredis` configurado especificamente pra BullMQ (`maxRetriesPerRequest: null`).

Esta story:
1. Instala `bullmq`
2. Define **estrutura modular** de filas em `src/lib/queue/`
3. Cria **registro central de queues** (`queues.ts`) com 1 fila: `default`
4. Cria **registro de jobs** (`jobs.ts`) com 1 job: `ping` (apenas loga payload e retorna `{ pong: true, processed_at }`)
5. Refatora `workers/index.mjs` → `workers/index.ts` com Worker BullMQ real consumindo `default`
6. Cria endpoint admin `/api/admin/queue/enqueue-ping` pra enfileirar manualmente (smoke)
7. Suite integration valida enqueue → process → result

**Padrão estabelecido:** próximas stories (3.5, 4.3, 7.2, 7.5) só adicionam novo arquivo em `src/lib/queue/jobs/` + novo case no Worker.

**Heads-up da Quinn (1.6):** Workers não têm auth Clerk. Quando jobs precisarem de tenant context (extractLabel, sendWhatsApp), o **payload deve carregar `condominioId`** e o processor usa `withTenantContext({ kind: 'tenant', condominioId, ... }, async (tx) => {...})` direto — NÃO `withTenant` (que depende de request).

**Out of scope:**
- Jobs reais (extractLabel=3.5, sendWhatsApp=4.3) — só o `ping` aqui
- BullMQ Dashboard / Bull Board — pós-MVP (ferramenta visual de admin)
- Cron jobs (BullMQ repeatable) — quando precisarmos (story 7.5 = expirar códigos ML após 30d)
- Múltiplas filas com prioridade — começamos com 1 (`default`)

## Notas para o @dev

### Decisões pré-aprovadas

- **`bullmq` (não Bull v4 legado):** BullMQ é o sucessor mantido ativamente, melhor TypeScript, suporta features modernas (flow producer, etc.).
- **1 fila `default` no MVP:** Refatoração pra múltiplas filas com prioridade vem quando virar dor (ex: separar fila de notificação WhatsApp da fila de IA pra evitar bloqueio mútuo).
- **Worker rodando em processo separado:** mantém isolamento da app Next.js. Crash do worker não derruba app e vice-versa.
- **`tsx` em vez de `node + .mjs`:** já temos tsx instalado (1.3 — Prisma seed). Permite `.ts` direto no worker. Em prod, story 8.4 ajusta pra `node` em build compilado.
- **Job `ping` retorna timestamp:** facilita testar latência end-to-end de enqueue → processamento.

### Pontos de atenção

1. **`maxRetriesPerRequest: null` no ioredis** já está configurado (story 1.7). É **requirement** do BullMQ — sem isso, lança warning e jobs podem perder.

2. **`removeOnComplete` é importante.** Sem isso, Redis vira lixo crescente. Default `{ age: 3600, count: 100 }` mantém últimos 100 jobs por 1h.

3. **Worker precisa do mesmo `redis` client da app:** importa de `@/lib/redis`. Pool compartilhado evita N conexões Redis.

4. **`worker.close()` é assíncrono e demorado** (espera jobs em-vôo terminar). Crucial em SIGTERM pra não perder trabalho. Adicionar timeout (ex: 30s) e force-quit se exceder.

5. **Job `name` é a chave do dispatcher.** `await defaultQueue.add('ping', payload)` → worker pega `job.name === 'ping'` → chama `processPing`. Erro de digitação no name = job vai pra failed silenciosamente.

6. **Tipagem dos payloads é crítica.** Use generics: `Job<PingPayload, PingResult, 'ping'>`. Sem tipos, refactor é arriscado.

7. **Endpoint admin requer super_admin** — porteiro normal não deve enfileirar jobs aleatoriamente (poderia drenar recursos). Em prod, esse endpoint nem deveria existir — só pra dev/debug.

8. **Path resolution `@/...` no worker:** worker está em `workers/`, fora de `src/`. tsconfig `paths` aceita ambos? Verificar. Se não, usar caminho relativo `../src/lib/queue/...` (feio mas funciona) ou ajustar tsconfig.

9. **Tests integration: criar Worker em-process.** Não depende do container worker. Padrão BullMQ:
   ```typescript
   const queue = new Queue('test-default', { connection: redis });
   const worker = new Worker('test-default', processorFn, { connection: redis });
   await queue.add('ping', payload);
   await new Promise(r => worker.once('completed', r));
   ```

### Tenant context em jobs (heads-up da Quinn 1.6)

Quando job precisar de tenant (extractLabel=3.5 vai precisar pra escrever em `pacote.ia_extracao_raw`):

```typescript
// Payload SEMPRE carrega condominioId
type ExtractLabelPayload = { pacoteId: string; condominioId: string; ... };

// Processor usa withTenantContext (NÃO withTenant — não tem auth context)
async function processExtractLabel(job: Job<ExtractLabelPayload>) {
  const ctx: TenantContext = {
    kind: 'tenant',
    userId: 'system', // jobs não tem user humano
    condominioId: job.data.condominioId,
    role: 'admin', // jobs operam com privilégio admin do tenant
  };
  return withTenantContext(ctx, async (tx) => {
    // queries Prisma filtradas por RLS automaticamente
  });
}
```

Job `ping` desta story **NÃO precisa de tenant** (apenas eco), mas vou documentar o pattern no `jobs.md` runbook pra próximas stories.

## Dev Notes

### Decisões e descobertas

**1. REDIS_URL no `.env.local` mudado pra `localhost` (mesma issue do DATABASE_URL)**
Tests rodam no host (não no container) → não conseguem resolver `redis:6379`. Solução já estabelecida na 1.3 pra Postgres: `.env.local` usa `localhost`, compose tem override `redis:6379` hardcoded no anchor `x-app-env`. Apliquei mesma estratégia. Pattern consistente.

**2. T8 reescrito como `tsx workers/index.ts`** (script `worker:dev`)
`tsx` já estava instalado (1.3 — Prisma seed). Permite usar `.ts` direto no worker sem build. Em prod (story 8.4) será `node workers/index.js` em build compilado.

**3. Healthcheck Docker do worker — pattern genérico**
Originalmente `pgrep -f "node workers/index"` (1.7 — node mjs). Agora é `tsx workers/index.ts`. Solução: `pgrep -f "workers/index"` casa **ambos** padrões — funciona pra story 1.2 placeholder (node mjs) e 1.8 atual (tsx ts) sem mudar quando refatorar pra build em 8.4.

**4. JOB_PROCESSORS map é mutável (`Record<string, AnyProcessor>`)**
Trade-off: type safety no payload do job individual (cada `processFn` tem `Job<TPayload>`), mas o map é genérico. Próximas stories (3.5, 4.3) só adicionam entrada — sem refactor do dispatcher. Aceitável pro MVP.

**5. Worker dispatcher lança erro pra job desconhecido**
`No processor registered for job name "<x>"`. Vai pra `failed` BullMQ após retries. Padrão defensivo: sysadmin investiga ao invés de job ficar "perdido" silenciosamente.

**6. Graceful shutdown com timeout 30s**
`worker.close()` espera jobs em-vôo terminarem. Sem timeout, deploy travaria indefinidamente se job lento. 30s é compromise — força exit se exceder. Em story 8.4, ajustar baseado em jobs mais longos (extractLabel pode levar 5-10s).

**7. Endpoint admin requer super_admin**
`/api/admin/queue/enqueue-ping` valida `getTenantContext().kind === 'super_admin'` — porteiro/admin normal recebe `UnauthorizedError`. Em prod, esse endpoint nem deveria existir (só dev/debug). Story 8.4 deve adicionar guard `NODE_ENV !== 'production'` ou remover.

**8. `process-incoming-msg` da story 7.2 vai precisar de role `webhook_worker`**
Anotei em `jobs.md` runbook: webhook handler de mensagens inbound precisa fazer lookup global por telefone (cross-tenant). Job correspondente também — usar BullMQ `connection` com `webhook_worker` role (BYPASSRLS) ou query via SQL raw. Decisão fica pra story 7.2.

**9. Test issue Vitest `it.skipIf` (5ª ocorrência)**
Mesma armadilha de 1.4, 1.6, 1.7 — `it.skipIf` é static-time. Solução: `describe.skipIf(STATIC_FLAG)` + `if (!RUNTIME_FLAG) return` dentro de cada teste quando precisa de runtime check. Padrão consistente em queue.test.ts.

**10. BullMQ `obliterate({ force: true })` no afterAll dos testes**
Limpa todos os jobs da fila de teste (incluindo failed). Sem isso, fica lixo no Redis entre runs. `obliterate` é a forma correta de "drenar" — `clean` só remove um estado por vez.
