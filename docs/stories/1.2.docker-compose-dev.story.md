## User Story

**Como** desenvolvedor do projeto,
**eu quero** um stack Docker Compose local com PostgreSQL 16, Redis 7, app Next.js e processo worker subindo com `docker compose up`,
**para que** eu possa rodar todo o ambiente de desenvolvimento isoladamente, sem instalar Postgres/Redis no host, e replicar o comportamento de produção desde o dia 1.

---

## Contexto

Esta story prepara a infraestrutura de desenvolvimento conforme `docs/architecture/ARCHITECTURE.md` §6.1. O objetivo é ter **paridade dev↔prod**: o mesmo `docker-compose.yml` (com pequenas diferenças no `.prod`) sobe local e na VPS Hostinger.

Ainda **não inclui**: Prisma migrations (story 1.3), Clerk (1.5), BullMQ jobs reais (1.8), Caddy (story 8.4 deploy). Só a fundação que permite as próximas stories conectarem em `postgres://postgres` e `redis://redis`.

**Pré-requisito do host:** Docker Desktop (ou Colima/OrbStack) instalado no macOS.

## Notas para o @dev

### Pontos de atenção

1. **Volume de `node_modules` separado** — bind mount do código-fonte combinado com `node_modules` do host (que é macOS) quebra binários compilados (sharp, esbuild). Solução: declarar volume nomeado `node_modules:/app/node_modules` que sobrescreve o bind. Mesmo padrão para `.next:/app/.next`.

2. **`depends_on` com healthcheck** — Compose v2 suporta `condition: service_healthy`. Sem isso, o `app` pode tentar conectar antes do Postgres estar pronto e crashar.

3. **Hot reload** — bind mount de `./` em `/app` permite que mudanças no código triggem o `next dev` sem rebuild da imagem. Não precisa fazer nada além do bind.

4. **Permissões macOS** — Docker Desktop no macOS faz file sharing via osxfs. Pode ficar lento. Se notar lentidão extrema, sugerir usuário trocar pra OrbStack/Colima (mais rápido). Não bloquear na story por causa disso.

5. **`worker:dev` é placeholder** — não implementar BullMQ aqui. Só garantir que o processo sobe e fica vivo. BullMQ vem na story 1.8.

6. **Não criar Caddy** — proxy reverso é da story 8.4 (deploy). Em dev, app expõe :3000 direto.

7. **Se Docker Desktop não estiver instalado no host**, parar e pedir pro usuário instalar antes de prosseguir. Não tentar instalar via brew sem confirmação (mudança no host).

### Decisões pré-aprovadas (não pedir confirmação)

- Postgres password = `dev_secret` em dev (em prod, story 8.4 usa Docker secrets)
- Sem PgAdmin/RedisInsight como containers extras — usar `docker compose exec` direto
- Sem rede customizada — Compose cria default network e basta

## Dev Notes

### Decisões e desvios do plano original

**1. Node 22-alpine confirmado em produção**
A story spec dizia Node 22 LTS no Dockerfile. Mantive `node:22-alpine` mesmo sabendo que o host roda Node 25 — paridade dev↔prod manda usar 22 dentro do container. Quality gates locais (que rodam no host com Node 25) continuam verdes.

**2. `tini` como init process (PID 1)**
Adicionei `tini` em todos os stages (`apk add --no-cache tini`). Razão: `npm` e `node` como PID 1 não respondem corretamente a SIGTERM, gerando shutdowns sujos e zombie processes. Padrão amplamente recomendado pra containers Node em produção. Custo: ~200KB na imagem. Vantagem: `docker compose down` é instantâneo, sem SIGKILL forçado.

**3. Stage `production` já preparado mas não usado nesta story**
Implementei os 4 stages completos (deps, development, builder, production) com USER não-root + `output: standalone`. Story 1.2 usa só `development`, mas deixar pronto economiza retrabalho na story 8.4 (deploy VPS). Nota: `next.config.ts` precisa ganhar `output: 'standalone'` quando 8.4 chegar — anotei como gotcha.

**4. Logs vazios no `docker compose logs` — não-bloqueante**
Após boot, `docker logs` ficou vazio mesmo com Next.js claramente respondendo HTTP 200 (validei via curl direto + `ps aux` dentro do container). Causa provável: combinação de `npm` + `tini` em modo non-TTY bufferizando stdout. Funcionalidade NÃO afetada — apenas observabilidade em dev. Workaround se incomodar:
```bash
# adicionar ao service `app`/`worker` do compose
tty: true
stdin_open: true
```
Não apliquei agora porque (a) não bloqueia nada e (b) prefiro investigar mais antes de mudar config. Registrei no backlog.

**5. Bind mount + volumes nomeados separados — funcionou perfeitamente**
A combinação `../..:/app` (bind do código) + `node_modules:/app/node_modules` + `next_cache:/app/.next` (volumes nomeados) sobrescreveu corretamente os subpaths. Validado no smoke test: container vê código com hot reload, mas usa `node_modules` linux-x64 instalado na imagem (não os do macOS). Padrão é descrito explicitamente no compose para documentar a decisão.

**6. Path do Google Drive funcionou com Docker mesmo com espaços**
Preocupação inicial: o caminho `Meu Drive/myworkspace/...` tem espaços. Docker lidou bem — bind mount funcionou, `docker compose` processou o path sem escapes manuais. Crédito ao OrbStack (file sharing nativo, sem osxfs lento).

**7. `name: gestao-pacotes-dev` no compose**
Adicionei `name:` no topo pra fixar o prefixo dos containers (`gestao-pacotes-dev-app-1` etc). Sem isso, Compose usa o nome do diretório (`gestao_Pacotes_Condominios` — feio em logs). Convenção: `gestao-pacotes-dev` em dev, `gestao-pacotes-prod` em prod (story 8.4).

**8. `infra/README.md` ficou denso (200+ linhas)**
Optei por runbook completo em vez de README mínimo. Razão: este arquivo será consultado em momentos de stress (deploy quebrou, container não sobe). Investimento agora paga muito dividendo depois. Inclui troubleshooting de portas ocupadas, volumes corrompidos, OOM, build lento.

**9. `.dockerignore` agressivo**
Excluí `infra/`, `docs/`, `prisma/`, todos os `.cli/` (codex, claude, cursor, etc.) e o framework `.aiox-core/`. Resultado: contexto Docker enviado pro daemon ficou pequeno (~150MB em vez de ~1.2GB se incluísse tudo). Build-time muito mais rápido.
