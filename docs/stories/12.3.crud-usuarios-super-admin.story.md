## User Story

**Como** super-admin,
**eu quero** listar todos os usuarios do sistema com filtros (role, condominio, status), editar role/nome, e desativar/reativar usuarios,
**para que** eu possa gerenciar o acesso de todos os condominios a partir de um unico painel.

## Acceptance Criteria

- [x] AC1: Pagina `/super-admin/users` lista TODOS os usuarios (admin_master, admin_funcionario, porteiro) cross-tenant
  - Colunas: Nome, Email, Role (badge), Condominio, Status (Ativo/Pendente/Inativo), Acoes
  - Paginacao server-side (20 por pagina)
- [x] AC2: Filtros funcionais na listagem
  - Filtro por role (select: todos, admin_master, admin_funcionario, porteiro)
  - Filtro por condominio (select com nome dos condominios ativos)
  - Filtro por status (todos, ativo, pendente, inativo)
  - Busca textual por nome/email
- [x] AC3: Super-admin pode editar role e nome de qualquer usuario
  - PATCH `/api/super-admin/users/[id]` com body `{ nome?, role? }`
  - Nao pode mudar role para super_admin (guard)
  - Nao pode editar a si mesmo (guard)
- [x] AC4: Super-admin pode desativar/reativar usuario
  - PATCH `/api/super-admin/users/[id]` com body `{ ativo: false }` / `{ ativo: true }`
  - Usuario desativado perde acesso no proximo request (user.ativo check ja existe no Clerk webhook)
- [x] AC5: Badges visuais corretos
  - "Pendente" (amarelo) — clerk_id comeca com `pending_clerk_link_`
  - "Ativo" (verde) — ativo=true e clerk_id real
  - "Inativo" (cinza) — ativo=false
- [x] AC6: Criar usuario continua funcionando (form existente do 8.5 integrado na mesma pagina)

## Decisoes tecnicas

1. **Reusar `AdminsListClient`:** Expandir para aceitar todos os roles (renomear para `UsersListClient` ou adicionar prop)
2. **API unificada:** Endpoint PATCH `/api/super-admin/users/[id]` para editar + desativar (mesma rota, campo diferente)
3. **Listagem:** Novo helper `listAllUsers(filters)` que aceita paginacao + filtros
4. **Edit inline:** Sheet lateral com form de edicao (nome + role select)
5. **Nao precisa deletar:** Soft delete = `ativo: false`. Sem hard delete.

## Out of scope

- Deletar usuario permanentemente (apenas desativar)
- Trocar email (exige fluxo Clerk separado)
- Mover usuario entre condominios (complexo, story dedicada)
- Bulk operations (desativar todos de um condo — usar 12.2 pra isso)
