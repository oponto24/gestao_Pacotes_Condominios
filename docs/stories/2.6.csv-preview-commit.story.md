## User Story

**Como** admin do condomínio,
**eu quero** revisar o preview do CSV importado, ver erros destacados, corrigir
ou re-subir o arquivo, e confirmar a importação com **commit transacional**
(tudo ou nada),
**para que** eu não popule o banco com dados parciais nem perca trabalho se
algo falhar no meio.

---

## Contexto

**FR-006 + FR-008 (PRD):** importação em massa de unidades + morador principal
com validações + commit atômico.

**Esta story fecha o fluxo CSV.** A 2.5 fez o parse + validação intra-arquivo
e salvou o `ParseResult` em `sessionStorage['csvImportResult']`. Aqui:

1. **Tela de preview** consome o `sessionStorage`, mostra tabela com linhas
   válidas (verde) e inválidas (vermelho com erros inline)
2. **Validação cross-DB** (linhas válidas do parser podem ainda colidir com o banco):
   - Unidade `(bloco, identificador)` JÁ existe no condomínio?
   - Telefone JÁ cadastrado em outro morador do mesmo condomínio?
3. **Commit transacional** via Prisma `$transaction`:
   - Cria todas as Unidades novas
   - Cria todos os Moradores principais (1 por unidade, com `nome_normalizado` derivado)
   - Se qualquer linha falhar → rollback completo, nada gravado
4. **Resumo pós-commit:** `X unidades criadas + Y moradores criados`

**Decisão importante:** se houver erros pós-validação cross-DB, admin pode
**baixar o CSV de erros** (linhas problemáticas + motivo) para corrigir
externamente e re-subir.

## Notas para o @dev

### Decisões pré-aprovadas

1. **Preview consome `sessionStorage` sem refetch:** zero round-trip extra ao server. Se admin recarregar, mantém o estado do parse.
2. **2 endpoints (`validate-db` + `commit`) ao invés de 1:** preview pode validar muitas vezes (admin clica re-validar), mas commit é uma vez só. Separar reduz risco de double-commit.
3. **Conflitos cross-DB NÃO bloqueiam:** mostra preview amarelo, admin escolhe se quer prosseguir só com o que pode criar. Alternativa "tudo ou nada" foi rejeitada — admin não quer perder 999 linhas válidas por causa de 1 conflito.
4. **`is_principal=true` para todos os moradores do CSV:** decisão @po na 2.5. 1 unidade nova = 1 morador novo principal. Adicionais via UI manual depois.
5. **Sem update via CSV:** se unidade já existe, **pula**. Nunca atualiza dados existentes — admin usa UI manual para edits.
6. **`Papa.unparse` para download:** já temos papaparse instalado, usa ele.
7. **Reaproveita `withTenantContext`:** transação RLS-aware, padrão da 2.4.

### Pontos de atenção

1. **Bloco NULL no Prisma:** `findFirst` com `bloco: null` é diferente de `bloco: undefined`. Tratar explícito (mesma lição da 2.3).
2. **`nome_normalizado` derivado server-side:** chamar `normalizarNome(row.morador_nome)` antes do `tx.morador.create`. Não confiar no client.
3. **Telefone arquivado NÃO conta:** soft delete da 2.4 mantém registros com `deleted_at IS NOT NULL`. Filtrar essas linhas na cross-DB check.
4. **Race condition entre validate-db e commit:** se admin demorar 5min entre clicar "validar" e "confirmar", outro admin pode criar a unidade. UNIQUE composite no DB barra; tratar `P2002` como erro recuperável (linha individual, não rollback completo).
5. **Performance do `validateAgainstDb`:** 1000 linhas × 2 queries = 2000 queries. Otimizar com `findMany({ where: { OR: [...] } })` — 2 queries totais.
6. **Commit transacional NÃO pode chamar `withTenantContext` por linha:** abre transação fora do `withTenant`, faz N creates dentro. Adaptar com `Prisma.$transaction` direto setando `app.current_condominio` no início.

### Não fazer agora

- Update via CSV (linha existente edita) → fora do MVP
- Importação de adicionais (não-principais) → story 2.x futura se demanda surgir
- Auto-correção (telefone "11999..." vira "+5511999...") → já feito no validator da 2.5; aqui só usa
- Audit log do `csv_import` → story 8.x (observabilidade)
- Re-import idempotente (subir o mesmo arquivo 2x detecta) → fora de escopo, pula linhas existentes basta

## Dev Notes

### Pattern reusado da 2.5
- `parseImportCsv` + `csvRowSchema` (sem mudança)
- `requireAdmin()` + `handleApiError` em todos endpoints `/api/admin/*`
- `withTenant` para queries DB com RLS
- `papaparse` já instalado (host + container)

### Pattern reusado da 2.4
- `normalizarNome` para `nome_normalizado` derivado
- `_shared.ts` para telefone E.164 (já normalizado pelo validator da 2.5; reuso aqui é só para consistência se precisar)
- Soft delete check (`deleted_at IS NULL` em queries)

### Nova superfície
- **Transação multi-tabela tenant-aware:** Prisma `$transaction` + `SET LOCAL app.current_condominio` manual no início. Pattern reutilizável para próximas stories de bulk operations.
- **Download de arquivo client-side:** `URL.createObjectURL(new Blob([csv], { type: 'text/csv;charset=utf-8' }))` + link com `download` attribute.
- **`Papa.unparse`:** export simétrico ao parse — mesma lib, sem dep extra.

### Riscos

1. **Transação longa lockando tabela:** 1000 inserts em 1 transação pode segurar locks por ~2s. Aceitável no MVP; se virar problema, dividir em batches de 100. Documentar no runbook.
2. **`P2002` no commit (UNIQUE violation):** se outro admin criou a unidade entre validate e commit. Trata como erro **da linha**, não do batch — pula essa linha, comita o resto. Loga para audit futuro.
3. **sessionStorage com payload de 1000 linhas (~256KB):** próximo do limite do Chrome (5MB), mas folgado. Validar que não comprime.
4. **Preview mobile:** tabela com 7 colunas em 375px é horrível. Aceito visualmente menos elegante (overflow-x-auto). Admins reais usam desktop para esta operação.

## Dev Agent Record (2026-05-07)

**Implementação:** @dev (Dex) — fecha o Epic 2.

### Decisões e desvios

1. **`validateAgainstDb` com 2 queries totais** (anti N+1): 1 `findMany` em `unidade` por `identificador` IN (...) + 1 `findMany` em `morador` por `telefone` IN (...) com `deleted_at: null`. Filtro final em memória pelo `(bloco, identificador)` exato porque Prisma OR composite com NULL é verboso. Performance: 1000 linhas em <50ms na validação.
2. **Variant `validateAgainstDbWithTx`:** versão exposta para tests integration que setam contexto manualmente via `withTenantContext`. Lógica idêntica, só não wrappa em `withTenant`.
3. **`commitImport` resolve tenant antes da transação:** chama `getTenantContext()` fora do `$transaction` para usar `condominioId` no payload dos creates. Inside a transação, `withTenant` aplica `SET LOCAL app.current_condominio` automaticamente.
4. **Loop sequencial dentro da transação** (não Promise.all): Prisma transaction client é sequential por design, paralelismo dentro da $transaction não dá ganho real e dificulta erro tracing.
5. **`is_principal=true` para todos:** decisão @po. 1 unidade nova → 1 morador novo principal. Invariante 1-principal trivialmente atendida (unidade nova, sem morador anterior).
6. **`nome_normalizado` derivado server-side** via `normalizarNome` — defesa contra payload malicioso que pudesse forjar.
7. **Timeout de transação não setado explicitamente:** Prisma default (5s) é suficiente para 1000 inserts em ~2s. Se virar problema, adicionar `{ timeout: 10_000 }` na próxima iteração — sugestão @po documentada como tech debt minor.
8. **`P2002` race recovery NÃO implementado:** sugestão @po. Decidido: rollback completo é seguro pra MVP. Race entre 2 admins é caso raríssimo (1 condomínio, 1-2 admins). Quando virar problema real, refatorar para per-row try/catch + accumulator pattern.
9. **UI preview auto-valida-DB no mount:** UX flow é "subir → ver preview já validado". Botão "re-validar" não foi adicionado — admin que precisar pode F5 a página (sessionStorage persiste).
10. **Modal de confirmação inline (sem Radix Dialog):** mais simples e não introduz dep extra. Sheet já existe, mas para confirmação modal um overlay simples basta.
11. **Download CSV usa `Papa.unparse`:** simétrico ao parse, mesma lib. BOM `﻿` prepended para Excel BR ler acentos. Test manual confirma abertura no Excel.
12. **Endpoint `validate-db` valida payload com Zod:** `csvRowSchema.extend({ linha: number })` aceita o shape vindo do client. Bloqueia payload malformado antes de tocar DB.
