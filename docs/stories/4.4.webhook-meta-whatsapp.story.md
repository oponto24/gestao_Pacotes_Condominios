## User Story

**Como** sistema,
**eu quero** receber webhooks da Meta com status de entrega (`sent/delivered/read/failed`) e mensagens inbound (respostas dos moradores),
**para que** o painel admin mostre status real de cada notificação e o sistema possa processar códigos ML (Epic 7).

---

## Contexto

**FR-036 (PRD):** "Sistema deve registrar status de entrega do WhatsApp (sent / delivered / read / failed) via webhook Meta."
**FR-052:** "Sistema deve identificar o morador remetente pelo número de telefone."
**NFR-024:** "Webhooks Meta validados por assinatura HMAC."

**Esta story entrega:**
- GET handler — verificação inicial Meta (challenge token)
- POST handler — recebe eventos, valida HMAC, processa async via fila
- Job `processWhatsappWebhook` que atualiza `WhatsAppMessage` ou cria mensagem inbound

**Tipos de evento Meta:**
- `messages` (status updates: sent/delivered/read/failed)
- `messages` (inbound: morador respondeu)

**Bloqueio Etapa 6 do runbook:** essa story precisa estar em produção pra Etapa 6 (configurar webhook no painel Meta com URL real). Por isso, smoke real só roda após deploy.

## Acceptance Criteria

### AC1 — Variável `META_WEBHOOK_VERIFY_TOKEN`
- [x] Adicionar em `.env.app.example`, `.env.prod.example`
- [x] Adicionar em `src/lib/env.ts` Zod schema (obrigatória se `META_DISABLED !== 'true'`)
- [x] Documentar no runbook como gerar: `openssl rand -hex 32`

### AC2 — GET handler — verificação Meta
- [x] Route `src/app/api/webhooks/meta-whatsapp/route.ts` — handler GET
- [x] Lê query params: `hub.mode`, `hub.verify_token`, `hub.challenge`
- [x] Se `mode === 'subscribe' && token === META_WEBHOOK_VERIFY_TOKEN` → `return new Response(challenge, { status: 200 })`
- [x] Senão → `return new Response('Forbidden', { status: 403 })`

### AC3 — POST handler — validação HMAC + enfileiramento
- [x] Lê header `x-hub-signature-256` (formato `sha256=...`)
- [x] Lê body bruto (`request.text()` ANTES de parsear)
- [x] Calcula HMAC-SHA256 com `META_APP_SECRET` sobre body bruto
- [x] Compara com timing-safe (`crypto.timingSafeEqual`)
- [x] Se inválido → 401 (log warning com fbtrace_id se houver)
- [x] Se válido:
  1. Parsear body como JSON
  2. Para cada `entry[].changes[].value` enfileirar job `processWhatsappWebhook` com payload
  3. Retornar `200 OK` em <1s (Meta exige resposta rápida — processamento real é assíncrono)

### AC4 — Job `processWhatsappWebhook`
Em `src/lib/queue/jobs/process-whatsapp-webhook.ts`:
- [x] Constante `PROCESS_WHATSAPP_WEBHOOK_JOB_NAME = 'processWhatsappWebhook'`
- [x] Payload: `{ value: MetaWebhookValue }` (tipo da story 4.1 ampliado)
- [x] **Usa conexão `DATABASE_WEBHOOK_URL`** (worker dedicado com BYPASSRLS, story 1.4)
- [x] Roteia entre dois sub-handlers conforme tipo:
  - `value.statuses[]` → `handleStatusUpdate`
  - `value.messages[]` → `handleInboundMessage`

### AC5 — `handleStatusUpdate`
Para cada status:
- [x] Lookup `WhatsAppMessage` por `meta_message_id = status.id`
- [x] Se não encontrado → log warning + skip (mensagem foi enviada por outro processo / mensagem teste)
- [x] Update conforme `status.status`:
  - `sent` → `status: sent, sent_at`
  - `delivered` → `status: delivered, delivered_at`
  - `read` → `status: read, read_at`
  - `failed` → `status: failed, failed_at, failure_reason: status.errors[0].title`

### AC6 — `handleInboundMessage`
Para cada mensagem:
- [x] Identificar condomínio: lookup `Morador` por `telefone = message.from`. Se único, usa `condominio_id` dele. Se múltiplo (morador em N condomínios), prioriza com mensagem outbound mais recente. Se zero, log + ignora.
- [x] Cria `WhatsAppMessage`:
  - `direction: inbound`, `meta_message_id: message.id`, `from_phone: message.from`, `to_phone: <numero compartilhado>`, `body_text: message.text.body`, `morador_id`, `condominio_id`
  - `status: delivered` (recebida)
- [x] **Não processa código ML aqui** — apenas registra. Epic 7 vai criar handler separado escutando essa criação.
- [x] Idempotência: se já existe `WhatsAppMessage` com `meta_message_id`, ignora (Meta pode reentregar)

### AC7 — Tests
- [x] `tests/unit/webhook-hmac.test.ts`:
  - HMAC válido → 202 + job enfileirado
  - HMAC inválido → 401
  - HMAC ausente → 401

### AC8 — Logging estruturado

## Riscos & mitigações

| Risco | Mitigação |
|-------|-----------|
| Meta reentrega evento → race condition | `meta_message_id @unique` no schema (já existe) garante idempotência |
| Body bruto consumido antes do parse → HMAC quebra | Ler `request.text()` UMA vez antes de parsear; reusar string |
| Worker offline → Meta timeout | POST handler retorna 200 imediatamente após enfileirar; processamento fica resiliente |
| `META_APP_SECRET` errado em prod → tudo cai em 401 | Smoke pós-deploy obrigatório antes de declarar story Done |
