## User Story

**Como** sistema,
**eu quero** receber webhooks da Meta com status de entrega (`sent/delivered/read/failed`) e mensagens inbound (respostas dos moradores),
**para que** o painel admin mostre status real de cada notificação e o sistema possa processar códigos ML (Epic 7).

---

## Contexto

**FR-036 (PRD):** "Sistema deve registrar status de entrega do WhatsApp (sent / delivered / read / failed) via webhook Meta."
**FR-052:** "Sistema deve identificar o morador remetente pelo número de telefone."
**NFR-024:** "Webhooks Meta validados por assinatura HMAC."

**Esta story entrega:**
- GET handler — verificação inicial Meta (challenge token)
- POST handler — recebe eventos, valida HMAC, processa async via fila
- Job `processWhatsappWebhook` que atualiza `WhatsAppMessage` ou cria mensagem inbound

**Tipos de evento Meta:**
- `messages` (status updates: sent/delivered/read/failed)
- `messages` (inbound: morador respondeu)

**Bloqueio Etapa 6 do runbook:** essa story precisa estar em produção pra Etapa 6 (configurar webhook no painel Meta com URL real). Por isso, smoke real só roda após deploy.

## Acceptance Criteria

### AC1 — Variável `META_WEBHOOK_VERIFY_TOKEN`
- [ ] Adicionar em `.env.app.example`, `.env.prod.example`
- [ ] Adicionar em `src/lib/env.ts` Zod schema (obrigatória se `META_DISABLED !== 'true'`)
- [ ] Documentar no runbook como gerar: `openssl rand -hex 32`

### AC2 — GET handler — verificação Meta
- [ ] Route `src/app/api/webhooks/meta-whatsapp/route.ts` — handler GET
- [ ] Lê query params: `hub.mode`, `hub.verify_token`, `hub.challenge`
- [ ] Se `mode === 'subscribe' && token === META_WEBHOOK_VERIFY_TOKEN` → `return new Response(challenge, { status: 200 })`
- [ ] Senão → `return new Response('Forbidden', { status: 403 })`

### AC3 — POST handler — validação HMAC + enfileiramento
- [ ] Lê header `x-hub-signature-256` (formato `sha256=...`)
- [ ] Lê body bruto (`request.text()` ANTES de parsear)
- [ ] Calcula HMAC-SHA256 com `META_APP_SECRET` sobre body bruto
- [ ] Compara com timing-safe (`crypto.timingSafeEqual`)
- [ ] Se inválido → 401 (log warning com fbtrace_id se houver)
- [ ] Se válido:
  1. Parsear body como JSON
  2. Para cada `entry[].changes[].value` enfileirar job `processWhatsappWebhook` com payload
  3. Retornar `200 OK` em <1s (Meta exige resposta rápida — processamento real é assíncrono)

### AC4 — Job `processWhatsappWebhook`
Em `src/lib/queue/jobs/process-whatsapp-webhook.ts`:
- [ ] Constante `PROCESS_WHATSAPP_WEBHOOK_JOB_NAME = 'processWhatsappWebhook'`
- [ ] Payload: `{ value: MetaWebhookValue }` (tipo da story 4.1 ampliado)
- [ ] **Usa conexão `DATABASE_WEBHOOK_URL`** (worker dedicado com BYPASSRLS, story 1.4)
- [ ] Roteia entre dois sub-handlers conforme tipo:
  - `value.statuses[]` → `handleStatusUpdate`
  - `value.messages[]` → `handleInboundMessage`

### AC5 — `handleStatusUpdate`
Para cada status:
- [ ] Lookup `WhatsAppMessage` por `meta_message_id = status.id`
- [ ] Se não encontrado → log warning + skip (mensagem foi enviada por outro processo / mensagem teste)
- [ ] Update conforme `status.status`:
  - `sent` → `status: sent, sent_at`
  - `delivered` → `status: delivered, delivered_at`
  - `read` → `status: read, read_at`
  - `failed` → `status: failed, failed_at, failure_reason: status.errors[0].title`

### AC6 — `handleInboundMessage`
Para cada mensagem:
- [ ] Identificar condomínio: lookup `Morador` por `telefone = message.from`. Se único, usa `condominio_id` dele. Se múltiplo (morador em N condomínios), prioriza com mensagem outbound mais recente. Se zero, log + ignora.
- [ ] Cria `WhatsAppMessage`:
  - `direction: inbound`, `meta_message_id: message.id`, `from_phone: message.from`, `to_phone: <numero compartilhado>`, `body_text: message.text.body`, `morador_id`, `condominio_id`
  - `status: delivered` (recebida)
- [ ] **Não processa código ML aqui** — apenas registra. Epic 7 vai criar handler separado escutando essa criação.
- [ ] Idempotência: se já existe `WhatsAppMessage` com `meta_message_id`, ignora (Meta pode reentregar)

### AC7 — Tests
- [ ] `tests/unit/webhook-hmac.test.ts`:
  - HMAC válido → 202 + job enfileirado
  - HMAC inválido → 401
  - HMAC ausente → 401
- [ ] `tests/unit/webhook-status-handler.test.ts`:
  - status `delivered` atualiza `WhatsAppMessage` correta
  - status `read` não regride pra delivered se já estava read (idempotência)
  - mensagem inexistente → log + skip
- [ ] `tests/unit/webhook-inbound-handler.test.ts`:
  - Inbound conhecido → cria mensagem com morador vinculado
  - Inbound de telefone desconhecido → cria mensagem com `morador_id = null`
  - Reentrega da Meta → não duplica
- [ ] Fixtures: `tests/fixtures/meta-webhooks/*.json` com payloads reais de exemplo Meta (copiar da doc Meta)

### AC8 — Logging estruturado
- [ ] Toda chamada loga: `entry_id`, `event_type`, `meta_message_id`, `processing_ms`
- [ ] Erros de HMAC logam apenas IP + tamanho do body (NUNCA logar headers de assinatura — vazaria info sensível)

### AC9 — Smoke manual (pós-deploy)
- [ ] Documentar no runbook Etapa 6: "Após deploy, configurar webhook no painel Meta com URL `https://condominios.oponto24.com.br/api/webhooks/meta-whatsapp` e token gerado em `META_WEBHOOK_VERIFY_TOKEN`. Meta faz GET de verificação na hora — confirmar 200 nos logs."

## Riscos & mitigações

| Risco | Mitigação |
|-------|-----------|
| Meta reentrega evento → race condition | `meta_message_id @unique` no schema (já existe) garante idempotência |
| Body bruto consumido antes do parse → HMAC quebra | Ler `request.text()` UMA vez antes de parsear; reusar string |
| Worker offline → Meta timeout | POST handler retorna 200 imediatamente após enfileirar; processamento fica resiliente |
| `META_APP_SECRET` errado em prod → tudo cai em 401 | Smoke pós-deploy obrigatório antes de declarar story Done |
