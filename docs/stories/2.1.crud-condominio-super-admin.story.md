## User Story

**Como** super-admin do SaaS,
**eu quero** uma UI para listar, criar, editar e arquivar condomínios clientes,
**para que** eu consiga onboarding um novo condomínio sem precisar de SQL manual e
acompanhe o portfólio de tenants ativos.

---

## Contexto

Story inaugural do Epic 2 (cadastros). O modelo `Condominio` já existe no schema
(story 1.3) com 12 campos + soft delete (`deleted_at`). Falta:

1. **UI super-admin** em `/super-admin/condominios` (lista + criar + editar)
2. **API REST** em `/api/admin/condominios/*` (5 endpoints — list, get, create, update, archive)
3. **Validação Zod** no payload (CNPJ, CEP, estado UF)
4. **Componentes shadcn** mínimos: `input`, `label`, `table`, `sheet` (form lateral)
5. **Guard super_admin** em todos os endpoints (reutiliza `getTenantContext().kind`)

**Acesso:** apenas `super_admin` (story 1.6 confirmou que admin/porteiro têm
`condominio_id` setado e nunca veem cross-tenant). Condominio NÃO é tenant-scoped —
super-admin opera em todas as linhas. Endpoints **não usam** `withTenant()`; usam
`db` direto (com guard explícito de `kind: 'super_admin'`).

**Soft delete:** `deleted_at` setado em vez de DELETE físico. Lista filtra por
`deleted_at IS NULL` por padrão; query param `?include_arquivados=true` mostra todos.

**Out of scope:**
- Permitir admin editar próprio condomínio (story dedicada se necessário)
- Importação CSV de condomínios em massa (não há demanda — admin faz manualmente)
- Branding/logo upload por condomínio (futuro)
- Histórico de mudanças (audit_log) — story 8.3
- Exportação CSV (futuro)
- Bulk actions (selecionar múltiplos) — over-engineering pra MVP
- Filtros avançados (cidade, estado, ativo) — paginação simples + busca por nome

## Notas para o @dev

### Decisões pré-aprovadas

- **Soft delete via `deleted_at`** (NÃO DELETE físico): preserva integridade FK em `pacote`/`pacote_evento`/`audit_log` históricos. Schema já tem o campo.
- **CNPJ unique global** (não scoped por tenant): se um CNPJ já existe arquivado, novo cadastro deve falhar com 409 (admin restaura). Evita duplicação acidental.
- **Lista inclui `_count.unidades` e `_count.moradores`**: lookahead útil pro super-admin priorizar atendimento. 1 query Prisma com `_count` (não N+1).
- **Validação Zod compartilhada client+server**: schema único em `src/lib/validators/condominio.ts`. Importado tanto pela API route quanto pelo form client.
- **shadcn manual (não CLI)**: padrão do projeto (story 1.1). Copiar do shadcn docs e adaptar `cn()` import.
- **Sem React Hook Form se for over-engineering**: form simples (9 campos), `useState` + Zod parse onSubmit é suficiente. Se virar feio, então RHF.
- **Sem SuperAdminLayout dedicado nesta story**: story 8.1 trata. Aqui usa root layout + breadcrumb simples no header.
- **`/super-admin/condominios`** vs `/admin/condominios`: namespace `/super-admin/*` separado pra deixar explícito o escopo cross-tenant.

### Pontos de atenção

1. **RLS bypass intencional:** Condominio NÃO tem policy RLS (verificado nos migrations da 1.4). Logo `db.condominio.*` chamado direto via `app_runtime` funciona sem `withTenant`. **Validar:** rodar query no test integration sem context tenant pra confirmar.

2. **Guard explícito em CADA endpoint:** copiar pattern de `/api/admin/queue/enqueue-ping` (story 1.8). NÃO confiar em middleware — o middleware da 1.6 só carrega tenant context, não bloqueia por role.

3. **Sheet vs Dialog vs página dedicada:** Sheet (lateral slide-in) é melhor pra forms longos e mantém contexto da lista visível. Decisão: usar Sheet.

4. **Validação CNPJ:** regex de formato (`XX.XXX.XXX/XXXX-XX` ou 14 dígitos puros) **NÃO valida dígitos verificadores**. Out of scope (lib `validation-br` é overkill pra MVP). Documentar em Dev Notes.

5. **Validação telefone:** aceita `+55XXXXXXXXXXX` (E.164) OU formato brasileiro `(XX) XXXXX-XXXX`. Normalizar para E.164 no save (story 4.1 vai precisar pra Meta API).

6. **`router.refresh()` após submit:** revalida server component da lista sem F5. Pattern Next 15 App Router.

7. **Tests integration usam DATABASE_URL direto** (não `withTenantContext`) — Condominio é cross-tenant. Pattern: `db.condominio.deleteMany({ where: { nome: { startsWith: 'TEST_' } } })` no `beforeEach`.

### Não fazer agora

- ❌ Bulk actions (selecionar múltiplos pra arquivar)
- ❌ Filtros avançados (cidade, estado, status combinado)
- ❌ Exportação CSV
- ❌ Histórico de mudanças (audit_log) — story 8.3
- ❌ Validação de dígito verificador CNPJ
